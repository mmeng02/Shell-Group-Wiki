Running a single simulation on multiple processors[edit]

Changing the job file[edit]

A sample job file is provided. Note that there are several changes from a normal job file:

#!/bin/bash
#
#$ -V
#$ -cwd
#$ -j y
#$ -S /bin/bash
#$ -N sample_MPI
#$ -pe ompi 4
date

source /share/apps/useamber12.sh
mpirun -n 4 $AMBERHOME/bin/sander.MPI -O -rem 0 -i prod.in -o prod.mdout -c equil.rst -r prod.rst -x prod.mdinfo -p diph.prmtop

date

Firstly, a line is added to the top of the jobfile:

#$ -pe ompi 4

This line tells the cluster to run a multiprocessor job in the ompi environment, and the number that follows specifies the number of processors to be used.

Secondly, a line has been added above the actual job submission to Amber:

source /share/apps/useamber12.sh

This ensures that Amber12 will be used instead of Amber9.

Finally, the line used to submit the job to Amber is a little different as well:

mpirun -n 4 $AMBERHOME/bin/sander.MPI -O -rem 0 -i prod.in -o prod.mdout -c equil.rst -r prod.rst -x prod.mdinfo -p diph.prmtop

the number after mpirun -n indicates how many processors are to be used. Note that it should match the number of processors indicated in the above line. $AMBERHOME/bin/sander.MPI calls multisander. The -rem 0 indicates that this is NOT an REMD run. All other specifications after -O are the same as a regular Amber job.

Running REMD with Amber[edit]

To run replica exchange in Amber, see this tutorial as a starting point. Note that every replica needs its own .mdin file, for both the equilibration and production stages. A single groupfile that specifies all input and output filenames is also needed for each run. A python script that will set up these files (called setup_equil.py and setup_prod.py) using a file containing replica temperatures can be found in zin/home/cask0/home/cmills. See the python script for instructions on changing the specifications of the simulation.

Changing the job file[edit]

Another sample job file is provided below. Note the changes that have been made.

#!/bin/bash
#
#$ -V
#$ -cwd
#$ -j y
#$ -S /bin/bash
#$ -N
#$ -pe ompi 20
date

source /share/apps/useamber12.sh
mpirun -np 20 $AMBERHOME/bin/sander.MPI -ng 20 -groupfile prod.groupfile

date

The first two changes are synonymous with the changes made to run a single Amber simulation on multiple processors. The line to submit a job to Amber is slightly different from the previous example, however:

mpirun -np 20 $AMBERHOME/bin/sander.MPI -ng 20 -groupfile prod.groupfile

The number following mpirun -np still specifies the number of processors, and it should match the number of processors specified above in the job file. Again, the $AMBERHOME/bin/sander.MPI indicates that multisander is to be used. However, because there are multiple input files to be read, the input to Amber now takes the form

-ng 20 -groupfile prod.groupfile

Where the number after -ng specifies the number of replicas, and the file name after -groupfile specifies the groupfile that is to be used for this particular simulation.
