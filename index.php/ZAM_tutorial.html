Single peptide runs[edit]

Basics[edit]

The following tutorial describes how to create a ZAM simulation with one or more copies (full fragments) of the same protein. It is usually a good idea to keep the commands involving a call to zam.py in a bash script so you can remember exactly what you did - an example such script is provided in /share/apps/setup_zamrun.sh.

First, you will need to make a new path for your ZAM run: 

mkdir pathname
cd pathname

Initialize the run by providing ZAM with the complete protein sequence: 

zam.py initializerun protein_sequence

Here, protein_sequence can be one-letter codes, like ARGA, or three-letter codes, like ALA ARG GLY ALA.

Create a new fragment that spans the full sequence: 

zam.py newfrag full

Each time you run this command, ZAM will create a separate directory for each path. The directory will be named like 'rX-Ya' where X and Y are the beginning and ending residue numbers. This is where ZAM will store all data for the fragment.

If you want more than one of the same full fragment (e.g., in order to have multiple trials so as to average for good statistical accuracy), simply run this command multiple times until you have the desired number of fragments. You can check the listing of fragments by using: 

zam.py show

You now want to tell ZAM how long to run the fragments that you created: 

zam.py fragrextime fragnum run_ps

Here, fragnum is the number of the fragment or fragments. You can find these numbers from the zam show command. fragnum can be either a single fragment number, like 0, or a range of fragment numbers, like 0-4, or all of the fragments, "all". The run_ps value is the number of picoseconds to run the simulation. Typically, for short peptides, a good default value is 20000 ps, or 20 ns.

You also want to specify the portion of the simulation to be used for analysis. Ideally, you want to give the system ample time to first equilibrate before the analysis is performed. 

zam.py fraganaltime fragnum analysis_ps

By default, ZAM will use the last analysis_ps ps of the run for the analysis that it performs. A good default value is 2000 ps, although 5000 might not be a bad choice for longer runs. Typically, longer analysis times will require more time to complete the analysis portion of the simulation.

It is also customary to increase the number of swaps per cycle in each fragment to 5 to enhance efficiency: 

 zam.py swapspercycle all 5

At this point, you want to apply any additional options that your run might require. A description of these is below.

So far, you have only set up the configuration of your run, but the run itself has not yet been started. To determine the maximum number of processors that you can use, type: 

zam.py estimatereplicas all

ZAM can use less than this number as well, and does not require the maximum. If your simulation has multiple fragments, it is more efficient to reduce the number of processors that you use to a fraction of this amount. Typically, you should allocate ~30 processors to a single run with a couple of peptide fragments.

To send your ZAM simulation to the job queue, you first need to create a job script that will tell the system how to do this. A template is available and you should copy this file to your current directory (do not modify the original!): 

cp /share/apps/zamjob.sh .

You want to edit this file to give your job a name that will show up when it is running in the queue: 

nano zamjob.sh

In the nano program, only change the text that reads YOUR_JOB_NAME_HERE. Do not change any other text, including the characters that come at the beginning of that line. Your job name must be one word (no spaces, although you can use underscores). The job name must also start with a letter and not a number. When you are done editing the script file, pres Ctrl-X to exit, making sure to save your changes when the prompt asks.

You are now ready to submit your job. Every ZAM run requires a head node in addition to the replica nodes that you estimated above. First submit the head node, 

qsub zamjob.sh

To check if it is running, look at the output of 

qs

A code of "r" means that it is running, while "qw" means that it is waiting in the queue.

After submitting the head node, submit the replica nodes. You could enter qsub zamjob.sh multiple times, but this could be tedious. Instead, use the following helper command: 

smartsubmit -m number_of_jobs zamjob.sh

where you replace number_of_jobs to your earlier choice (e.g., 30).

You can now monitor the progress of the jobs using: 

cat progress.txt
cat rXX-YY/progress.txt

Check the times at which these files are updated. The main progress.txt file should be updated every hour, while the fragment progress.txt files should be updated every couple of minutes. If they are not being updated, it is likely that your ZAM run died. You can check by doing qs to see if it is still running on the system.

If for some reason you need to stop your job, run the command 

zam.py stop

This will tell the head node to quit the run and safely release all of the other processors. The ZAM job will not stop immediately but may take 1-2 minutes for all of the jobs to end. You be able to see if your job is finished by running qs or qstat. You can then modify the ZAM settings and restart the run with 

zam.py restartrun

followed by resubmission of the jobs.


Command summary[edit]

Here's a quick summary of different commands.

  • qs - to see what's currently running/loading in the queue
  • ls - to see what's in your current file
  • qsub zamjob.sh - to submit the first run (which will become the head node)
  • smartsubmit -m 30 zamjob.sh - to submit the 30 jobs that will run replicas
  • cat progress.txt - check to see if there are any errors in your runs
  • zam fragrextime 0 ps - sets the simulation length of fragment 0 to ps picoseconds
  • zam fraganaltime 0 ps - sets the analysis of fragment 0 to analyze the last ps picoseconds of the run

Sequence Details[edit]

If your sequence contains a disulfide bond between two cysteines and you want to apply it to all subsequently created fragments, you should enter the following command immediately after the zam initializerun command. 

zam addglobaldisulfide number number

Here, number refers to where the cysteine residue is with respect to the beginning of the chain.

Alternatively, you can add disulfides on a fragment-by-fragment basis using 

zam adddisulfide FRAGNUM number number

If you only want to create a fragment that spans part of the sequence, 

zam newfrag beginning end

beginning and end refer to the numbering of the residues of the chain.

Caps[edit]

By default, full sequences are simulated without capping groups (so that the termini are charged), while fragments are simulated with caps (blocking chemical groups that make the termini uncharged). You can modify whether or not there are caps using any of 

zam fragcaps fragnum none
zam fragcaps fragnum both
zam fragcaps fragnum ncap
zam fragcaps fragnum ccap

Force field[edit]

To change the force field model for a fragment, use the following command: 

zam forcefield fragnum forcefield

Here, forcefield is one of a predetermined number of options. Note that this command allows one to change the forcefield as well as the implicit solvent model. To use ff96 with igb8 instead of the ZAM default igb5, simply set forcefield to ff96_igb8.

If new forcefields unknown to ZAM (but on the path of the amber version in use) are desired, add the following line to the zamsetup.py file: 

 import mdsim
 mdsim.ForceFieldOptions = ["ff94", "ff96", "ff99", "ff99SB", "ff03", 
 "ff99mod5", "ff03d", "ff99SBe", "ff12SB", "new", “forcefield”, “names”]

Then the forcefields added to this list may be called in the first command. Note, though, that the forcefield must be present as amberX/dat/leap/cmd/leaprc."forcefield", where X is the amber version used and "forcefield" could be something like ff96.

Extending runs[edit]

If a simulation has finished, and you want to run it for additional time, use 

zam restartfrag 0 'new_total_ps'
zam restartrun
qsub zamjob.sh
smartsubmit -m 'number_of_jobs' zamjob.sh

Meta-analysis[edit]

You can perform various analyses on your results to compare with the native structure. 

metaanalysis rXX-YY --analysis_name

There are different kinds of analyses that can be performed by specifying analysis_name, such as

  • clustanalysis
  • rmsdanalysis
  • ssanalysis

The rmsdanalysis requires a native structure file called native.pdb be placed in the simulation path.

Fragment runs[edit]

In some cases, you may wish to run every fragment of a given length from a very large protein. The procedure is similar to the instructions above for single peptide runs, starting with entering the full sequence for the protein in the zam initializerun command.

However, instead of running the zam newfrag command, you will want to enter the following, 

zam makeseedfrags seedlen seedgap

Here, seedlen is the length of a seed fragment while seedgap is the gap in sequence in between adjacent fragments. To make every 9-residue fragment, 

zam makeseedfrags 9 1

This command will automatically create multiple fragments. Once you have created all of these, you can adjust any settings as before. For example, you will want to increase the number of swaps per cycle to 5. You can apply this to all fragments using 

zam swapspercycle all 5

Dipeptide runs[edit]

In this tutorial, we will set up a ZAM dipeptide simulation using two copies of the same peptide.

Setting up the simulation[edit]

1. First, login to your home folder in zin. You will need to create one folder for each different sequence you are simulating. For this example, name the folder ~USER/peptides/STVIIE. Then change to that folder.

2. Next we need to initialize the ZAM run by telling it what sequence to run. If the peptide is capped, you can run: 

     zam initializerun ACE SER TYR VAL ILE ILE GLU NME ACE SER TYR VAL ILE ILE GLU NME

Here, the codes ACE and NME are the N-terminal and C-terminal capping groups that prevent the peptide termini from being charged. On the other hand, if the peptides are not capped, but zwitterionic, run: 

     zam initializerun NSER TYR VAL ILE ILE CGLU NSER TYR VAL ILE ILE CGLU

Here, the prefix of N before SER indicates it is a charged N-terminal Serine. Similarly, the prefix of C before GLU indicates a C-terminal charged Glutamate. Note that if you don't put either ACE, NME, or some N and C prefixes, ZAM will think that you are simulating one long chain, rather than two separate peptides.

3. Now that we've told ZAM what the sequence is, we need to create a fragment. We will just do one fragment, which is not really a fragment but the full sequence. To create a new fragment for the full sequence: 

     zam newfrag full

You can now list all of the fragments in your simulation by typing: 

     zam show

4. We need to tell the simulation how long to run our fragment. Let's try 20ns first. In ZAM, times are measured in ps, so we want a time of 20000ps. Note that from the show command, our fragment is fragment number 0. Thus we run the command: 

     zam fragrextime 0 20000

5. We also need to tell our simulation the range of temperatures to explore. We want our replicas to vary between 270 and 500 K, which is a typical range for peptides that results in an acceleration of the run: 

     zam fragtemps 0 270 500

6. To add the umbrella restraints to our run: 

     zam addumbrellarest 0 -1 -2

     zam umbrelladistances 0 span 7.5 70 5.0

This will add umbrella restraints between chains 1 and 2 (positive numbers would indicate residues, rather than chains) starting at a distance between the two peptides of 7.5 Angstroms, going all the way up to 70. Angstroms, separated by windows of 5. Angstroms. The umbrella numbering starts at zero, such that U0 in this case would have an equilibrium spacing of 7.5 Angstroms and U12 would have an equilibrium spacing of 67.5 Angstroms (the umbrellas do not reach 70., as the specified range of spacings is not divisible by the specified window width). Alternatively, you can specify bins of unequal width by specifically enumerating each umbrella equilibrium spacing individually: 

     zam umbrelladistances 0 3.0 4.5 6.0 9.0 12.0 16.0 20.0 25.0 30.0 40.0 50.0 60.0

This, for example, would set the umbrella configuration for fragment "0" (the first numerical argument), with U0 centered at 3.0 Angstroms, ..., and U11 centered at 60.0 Angstroms. The sampling width of each umbrella is then determined by proximity to adjacent umbrellas, setting umbrella potential force constants accordingly. The advantage to this is that you can modulate the umbrella locations and spacings based on your initial sampling with umbrellas of equal width, reducing the spacing where sampling is bad and increasing/maintaining the spacing where sampling is sufficient; this allows you to get an overall smoother and clearer potential of mean force in the post-processing routine later on.

7. Finally, we want to turn up the swapping a little bit. This is a technical detail, but it will likely accelerate convergence in these runs: 

     zam swapspercycle 0 5

Submitting, monitoring, and managing the run[edit]

1. In order to submit your run, you need to copy zamjob.sh to your ZAM simulation folder from the share/apps folder in the main directory, renaming it in your folder to job.sh: 

    cp /share/apps/zamjob.sh job.sh

2. Next, edit job.sh and change the text that says "PUT_JOB_NAME_HERE" to some name for your job, without spaces. For example, you could name the job STVIIE. Make sure not to delete anything else on that line, or in that file. Then save and close it out.

3. To get your job running you need to submit it the same number of times as the number of replicas plus one extra for the "head node" which will organize communication between all of the processors. To estimate the total number of replicas: 

    zam estimatereplicas 0

When you get this number, write it down.

4. Next, you will submit one job and allow it to look around and figure out it's the first processor and therefore should nominate itself as head node: 

    qsub job.sh

5. Now wait about 10-15 seconds for this job to go through. You can use the following commands to list all of the jobs on the cluster: 

    qs or qstat

The letter "r" means running, "q" means waiting for a free processor to clear up (queued), and "t" just about to start. What happens is the queuing system on the cluster will find a free processor on one of the compute nodes and send your run to that one. Otherwise, it waits until another job finishes and then starts.

6. Once you've waited long enough, you want to submit more jobs for all of the replicas. Submit the same number of jobs (not including the head node job) as the estimate returned by ZAM. You can do this by running "qsub job.sh" that number of times, or you can use a shortcut: 

    smartsubmit -m 30 job.sh

if you want to submit 30 jobs.

7. Now, the simulation should be running. Look in the fragment folder (e.g., r1-14a/) at the progress.txt file. It should be updated about every couple of minutes with a detailed list of the current status of the simulation, and when it will finish.

8. If for some reason you need to stop your job, run the command 

    zam stop

This will tell the head node to quit the run and safely release all of the other processors. The ZAM job will not stop immediately but may take 1-2 minutes for all of the jobs to end. You be able to see if your job is finished by running qs or qstat.

Performing Analysis on Completed Fragments[edit]

Using the zamdistances Command[edit]

1. Once the status of the fragment you wish to analyze changes from "grow" to "done", edit job.sh and replace the command zammain.py with the following command:


zamdistances.py --out=FILENAME.txt --basedist=0 --binddist=16 r1-8a/ -1 -2 3 68 0.5


Notes:

-You may replace FILENAME with a suitable name of your choice (e.g., zamdist).

-The basedist option sets the baseline for which the program computes energies, in this example it is >= 0 Å. The binddist option sets the maximum distance over which the binding energy is computed, in Å.

-r1-8a/ should be replaced with the name of the folder containing the analysis of your fragment.

-The (-1 -2) command indicates that the energies will be computed between two chains.

-(3 68) are the minimum and maximum distances over which the analysis will be performed.

-(0.5) specifies the bin width the program will compute (this value is usually set to 0.5).


Output Notes:

-Energies in the resulting output are in units of kcal (e.g., free energy is in kcal/mol)

Using the zamcluster Command[edit]

1. Edit job.sh to include the command:


zamcluster.py r1-8a/ --distfilter=-1,-2 --distrange=0-22 --prefix=FILENAME


Notes:

-The distfilter command indicates that the analysis will be performed between two chains.

-distrange specifies the distances over which the analysis is computed.

-You may replace FILENAME with a suitable name of your choice (e.g., zamclust).

Specifying a Reference Frame[edit]

If it is desired that only a specific number of frames of the analysis time be read with with the zamdistances or zamcluster commands, the following commands may be appended:


--nskip=18000 --nread=2000


Notes:

-nskip specifies the beginning frame of the analysis time (in picoseconds) to be used.

-nread specifies the length of analysis time to be used.

-runwham must be added to the zamdistances command to ensure that the analysis is re-run anew for each time period. The temperature of the replica of interest is specified after the equal sign.

Examples:

zamdistances.py --out=FILENAME.txt --runwham=270 --basedist=0 --binddist=16 --nskip=18000 --nread=2000 r1-8a/ -1 -2 3 68 0.5

zamcluster.py r1-8a/ --distfilter=-1,-2 --distrange=0-22 --nskip=18000 --nread=2000 --prefix=FILENAME
