Big overview[edit]

What is ZAM?[edit]

ZAM is a multifunctional software package for simulating polypeptides, using replica exchange molecular dynamics as the main sampling algorithm. ZAM starts out with a single master sequence. From this master sequence, any number of fragments can be created, spanning a continuous stretch of residues. When ZAM is run, it simulates each fragment in a separate replica exchange simulation, with automated analysis upon completion. An important feature of the analysis is clustering, which generates a number of cluster structures and their populations.

During a run, ZAM automatically manages multiple nodes in a parallel environment and dynamically assigns them to different fragments as needed. When fragments are finished, they can be used to generate new fragments: (1) new residues are added in extended form on the termini of each of the clustered structures ("growth"), or (2) the clustered structures or two or more fragments are combined together using a Monte-Carlo rigid body alignment program ("assembly"). In either case, these new structures are used as initial conformations in a new replica exchange simulation.

ZAM can be used for protein folding via the zipping and assembly approach. When stable contacts in short fragments have been identified, residue-residue restraints can be added to lock those contacts into place in subsequent runs. ZAM also contains a Bayesian inference system, which takes measured information about residue-residue contacts from fragment runs (like contact probability, potential of mean force, backbone entropy, etc) and produces a running contact map indicating the likelihood that particular residue contacts are native. Moreover, a fragment decision-making algorithm can be used to automatically determine what fragments are to be simulated next, following the most promising zipping and assembly folding pathways.

ZAM also permits a high level of user control, and has a number of additional features: (1) torsional restraints to enforce particular secondary structures or to hold parts of the backbone in a user-specified reference conformation, (2) anchoring restraints to keep parts of a structure in a particular conformation in 3-D space, (3) multiple chain capability, (4) umbrella sampling along the distance between two residues, (5) ability to scale restraints to different strengths across replicas, and (6) automated weighted histogram analysis, which automatically detects different restraints in different replicas and compensates.

Dependencies and installation[edit]

Almost all of ZAM is written in Python, with the exception of a few modules written in Fortran and compiled for Python using f2py (for speed purposes). A complete file list is given here. ZAM requires:

  • Python - main code interpreter, version 2.4.
  • AMBER - MD simulation package. Almost all AMBER-specific ZAM routines are in mdsim.py, mdtrj.py, and coords.py. Binaries called include tleap, sander, and ambpdb. ZAM has been tested with AMBER 8 and 9.
  • numpy - Library for array class and numerical algorithms in Python.

To install, one simply needs to place the ZAM python scripts in a common folder that is referenced in the PYTHONPATH environment variable. In addition, each of three fortran files need to be compiled by the program f2py (supplied with numpy) and placed in the same folder: geometrylib.f90, whamlib.f90, proteinlib.f90. The compilation command looks something like this: f2py -c -m proteinlib proteinlib.f90.

Basic information[edit]

A complete ZAM setup for one master sequence requires its own folder, and fragment subfolders will be created inside it. All modifications to this setup are done using the zam.py command line tool. zam.py contains many commands for creating, modifying and displaying fragment data. Typically the commands are something like

zam.py command 5

which will do something to fragment number 5. A brief summary of each command can be obtained by just typing in

zam.py

without arguments. More detailed explanations of all of the commands are found with

zam.py help

To start a ZAM setup, use the command:

zam.py initializerun SEQ

where SEQ is either the 1-letter or 3-letter master sequence of amino acids, from which fragments will be created. More details on fragment creation are given below. To see a summary of all fragments in the current setup, use the command:

zam.py show

Fragments are numbered starting from 0, but they also have names like 1-20a, which indicates that a fragment spans residues 1 through (including) 20. The subfolder for a fragment is just the letter r plus its name, like r1-20a. The r stands for "residues" (it was decided that a path should start with a letter, not number). The letter suffix on the end of a fragment name enables multiple fragments with the same residue range but potentially different simulation conditions (restraints, etc). The letter starts at a, then goes to z, then aa, ab, ac, etc. Residue numbers always start at 1.

All modifications to a ZAM setup must be performed when ZAM is not running. These modifications are saved in a file called restart.dat. Do not delete this file! Once the desired setup has been achieved, a run is started in the following way: a job script file is created for a single-processor job, which contains the command zammain.py somewhere in it. This job file must change to the path of the ZAM run (where restart.dat is located) before zammain.py is executed.

To start a ZAM run, you submit the job the same number of times as the desired number of processors. The first job to execute will self-designate as the head node and will write a file called server.txt in the ZAM folder. Subsequent jobs to start will find this file and connect to the head node. Note that the head node does not actually run any replicas, but instead is responsible for task allocation and decision making. With ZAM, new processors can be added or deleted at any time during the run by submitting more jobs or deleting old ones (except the head node). ZAM dynamically allocates the workload amongst the client processors; it also will not accept more clients than it needs, and extraneous clients will simply terminate upon the beginning of job execution.

The history of clients connecting to the head node can be found in the file sockets.txt. The history of the specific actions ZAM is pursuing can be found in the file progress.txt, including time remaining estimates every hour. More detailed replica exchange information is found in the file progress.txt in each of the fragment subfolders, which is updated every swap cycle.

Normally ZAM will run until all fragments and analysis are finished, and then all jobs will terminate, including the head node. However, ZAM has restart capability and can be interrupted. To stop a run, one simply kills all the corresponding jobs. Then to restart a run, one must first delete the file server.txt before submitting new jobs. Otherwise, submissions will look for a nonexistent head node. The command

zam.py restartrun

will delete server.txt for you, if you prefer instead. When new jobs are submitted, ZAM will start the simulation exactly where it left off, and incorporate any modifications made during downtime using zam.py. The server.txt file must always be deleted before submitting a new job (i.e., where a new head node is required), even if the previous ZAM run finished on its own.

Fragments[edit]

Fragment stage[edit]

Fragments each have a "Stage" which contains a description of the current status of that fragment. "Stage" can take on the following values:

  • grow = fragment is new or has been grown from a previous fragment and is active for simulation
  • assem = fragment has been assembled from previous fragments and is active for simulation
  • done = fragment has finished being simulated

That is to say, any fragment marked grow or assem will be simulated when the ZAM run is started. Fragments marked done remain on disk and may be used later as parents of subsequent growth or assembly phases, but nothing will happen to them when a ZAM run is started. The current only difference between grow and assem is the initial structure generation: for grow, new residues are added in extended form on the end of the fragment; for assem, two fragments are combined using rigid body alignments in a Monte Carlo simulation.

Creating fragments[edit]

Once a master sequence has been specified using zam.py initializerun, fragments can be created in a number of ways. The simplest fragment is simply a stretch of residues started from extended conformation. For example, the command

zam.py newfrag 1 20

will create a fragment from residues 1-20. Another common procedure is to grow a fragment from an existing fragment. The command

zam.py growfrag 5 2

will create a new fragment grown from fragment number 5 by adding two residues on each end. Note that fragment 5 is unaffected; a new fragment is created instead.

To create a new fragment from scratch corresponding to the full (untruncated) sequence:

zam.py newfrag full

To assemble two or more fragments together, two steps are needed. First, you use the command

zam.py newfrag 1 20 6,7

to create a new fragment in residues 1-20 that uses structures from fragments 6 and 7. Notice the only difference here is the addition of the 6,7 term, which provides the numbers of the parent fragments. Also note that a new fragment is created and 6 and 7 are unaffected. ZAM will automatically find alignments of the cluster structures of fragments 6 and 7 to the current sequence, when making initial structures. Second, you need to change the fragment stage to indicate an assembly so that the rigid-body Monte Carlo will be used to make initial structures. To do this, use the command

zam.py fragstage 8 assem

to change the stage of the newly created fragment 8 to assembly (as opposed to grow for growth).

By default, fragments which are not the entire sequence are capped using ACE and NME blocking residues. This can be overridden using the command zam.py fragcaps, described in detail below. If the entire sequence is simulated, ZAM will ask if you want to keep the caps when you create the fragment.

Many commands in ZAM can be applied to multiple fragments simultaneously. For these commands, when a fragment number is requested, you can indicate more than one fragment using several different notations: 1-8 indicates fragments 1 through and including 8; 1,2,6 indicates fragments 1, 2, and 6; and 1-5,8 indicates fragments 1 through 5 and 8. To see which commands allow multiple fragments to be specified, read the detailed descriptions in zam.py help.

Once fragments have been created, you can list them using the command

zam.py show

to get a table of fragments and their basic features. There are, however, many more variables associated with each fragment. To see all variables, use the command

zam.py showall .

Typically you will want to do this only for one or two fragments at a time, so

zam.py showall 5

will only show the data for fragment 5, for example. A third command,

zam.py showrest

can be used to show associated residue-residue restraints with each fragment.

Running fragments[edit]

ZAM automatically determines how many replicas are to be used for a fragment of a given length. The maximum total number of processors that ZAM can use is equal to the sum of the number of replicas for each active fragment. To estimate this number, use the command

zam.py estimatereplicas 6-9

to estimate the total number of replicas used for fragments 6 through 9. When submitting a job with multiple fragments, it is often efficient to submit about 80% of the maximum number of replicas. This is because, when there are exactly the same number of processors as replicas, there is some inefficiency--replicas at high temperatures typically take half as much time to run in a swap cycle as those at lower temperatures. Reducing the number of processors enables ZAM to reuse some of the nodes that finished quickly at high temperature to work on lower temperature replicas. When, there is only one active fragment, however, this approach does not work.

Note that the head node does not participate in fragment simulations directly. Therefore, the minimum number of processors that can be used for a ZAM run is 2 (one head, one client). The maximum is actually the maximum number of replicas plus one additional for the head node. If you are running a job with only a single active fragment, the fastest way for it to complete is to submit a number of jobs equal to the estimated number of replicas plus one additional.

Deleting fragments[edit]

A fragment and its associated directory structure can be deleted with the command

zam.py delfrag 5

to delete fragment 5. This action does not affect the names of other fragments, but it does affect their numbers.

What happens to a fragment during a run[edit]

When a ZAM run begins after a fragment has been newly created, the following sequence of events occurs:

  1. The replica exchange workspace is created in the work/ subdirectory of the fragment. The data/ subdirectory is also initialized.
  2. Any parent structures are located, including those from parent fragment cluster structures and any additional user-supplied structures (anything named seed*.pdb in the conf/ subdirectory). The sequences of structures are aligned to the fragment sequence. The alignments are saved to a file called conf/seedlog.txt subdirectory.
  3. Initial structures are generated for the run. In the case of growth, this step is very quick as residues are simply added. For assembly, flexible loop residues are identified and a short Monte Carlo program runs to generate rigid body alignments. Depending on the size of the protein, this can take from 5 minutes to an hour. The initial structures are prepared for AMBER and saved as conf/init*.pdb.
  4. Along with initial structure generation, any reservoir structures (for reservoir swapping) are preprocessed.
  5. If anchoring or reference restraints are active, ZAM identifies pdb files that contain the needed reference configuration(s). ZAM first looks for files named conf/ref*.pdb. If none are found, it then looks for the first conf/seed*.pdb file. If still none are found, ZAM raises an error. Otherwise, the reference files are prepared for AMBER and saved as conf/refinit01.pdb, and on up.
  6. The initial pdb files are assigned to replicas, starting with the lowest temperatures first and proceeding from init01.pdb on up. Reference pdb files are similarly assigned. If there are less pdb files than replicas, ZAM simply cycles through the list as many times as necessary.
  7. Any restraint terms are processed and added to the replicas.
  8. The replica structures are energy-minimized.
  9. Replica exchange is performed until the target duration is reached.
  10. Analysis routines are performed and the results are saved as various files in /anal.
  11. Analysis metrics are used to update the master residue-residue contact probability map, using a Bayesian system.
  12. Temporary files are cleaned up.
  13. The fragment stage is marked as done and ZAM stops working on it.
  14. ZAM continues until all fragments reach done.

In actuality, ZAM works by maintaining a task queue system. Each of the tasks described above is placed in a queue, for each fragment. When the head node detects an available client, it will send the first task in the queue to that client, and wait for the task to be completed before moving on to the next item for a fragment; however, it will allow other tasks from other fragments to proceed simultaneously. If a client has an error or dies, it will re-send a task to a new processor. In this way, ZAM is able to parallel process all fragments at the same time, and it is adaptive to the number of participating processors.

When a ZAM job is interrupted and stopped, and subsequently resubmitted, a fragment resumes at the point it left off. That is to say, that once initial structures have been created, they will not be created again in a restarted run unless the fragment is explicitly reinitialized to do so. A common user error occurs when a ZAM run is stopped and changes are made to a fragment that would require structures to be re-processed, but the user does not reinitialize the fragment as well.

Restarting and reinitializing[edit]

Once a fragment has finished and its stage changed to done, ZAM will no longer perform calculations on that fragment. There are several ways to re-run various parts of the fragment:

zam.py restartfrag 5 
will re-run just the analysis for fragment 5. This command will delete the current analysis files, change the fragment stage to grow, and set the simulation to be started at the analysis stage when it is run.
zam.py restartfrag 5 10000 
will re-run fragment 5 and extend the replica exchange out to 10000 ps (assume it has already completed 5000 ps of simulation). The number 10000 indicates the total, summed time desired, not the amount of new time. This command will delete the current analysis files, change the fragment stage to grow, and set the simulation to be started at the replica exchange stage when it is run.
zam.py reinitfrag 5 
will re-initialize fragment 5, deleting all existing replica exchange data and starting as if the fragment were newly created. Be careful with this command; it irreversibly deletes any existing replica exchange and analysis data. This command is the only way to get a fragment to re-process initial configurations.

Fragment analysis[edit]

A number of different analyses are performed on each fragment, and the results stored in the anal/ directory. These are by default performed for the final 1ns of a replica exchange run. The analyses include:

  • weighted histogram analysis (WHAM) to determine relative free energies and weights of configurations in each replica at the target reweighting temperature. WHAM accounts for possible varying restraints across replicas. In determining weights at the target temperature, WHAM by default removes the effects of restraints.
  • contact probability and potential of mean force (PMF) calculations for all residue-residue pairs
  • PMFs between the centroids of chains, if there are multiple
  • contact probabilities and PMFs for any umbrella-sampled residue pairs
  • a score metric for each residue pair, which is a linear combination of contact probability and PMF (this is legacy code, and this metric is no longer considered useful)
  • average energies, broken down by components, in each replica
  • average metrics for each replica, like Rg, number of salt and hydrophobic pairs, coordination, fraction of kinds of residues on the interior, etc.
  • swapped restraint analysis, which shows the average replica index of restraints if they are allowed to be swapped (an optional, experimental feature)
  • backbone mesostate analysis, which lumps each residue into one of three classes ("a"-alpha, "b"-beta, "l"-loop) and then calculates the dominant states
  • punch analysis to measure cooperativity in residue-residue contact formation
  • clustering to find dominant structures, along with metrics for these clusters

The times from a replica exchange simulation used for analysis can be modified:

zam.py fraganaltime 5 2000 
will modify fragment 5 to perform analysis on the last 2000 ps
zam.py fraganaltime 5 2000 1000 
will modify fragment 5 to perform analysis by skipping the first 1000 ps and using the next 2000 ps, rather than using the final 2000 ps
zam.py fraganaltime 5 auto 
will modify fragment 5 to use the default analysis times

Fragment directory structure[edit]

For each fragment, a directory structure is created to organize all of the information from the REMD run and analysis. The detailed file structure is:

  • rX-Y/
    • progress.txt - summary of replica exchange
    • anal/ - analysis directory
      • fraganal.dat.gz - analysis data pickled and compressed; used by ZAM for quick loading
      • whamresults.txt - free energies in each replica from WHAM analysis
      • contactprob.txt - contact probabilities between residue pairs
      • pmf.txt - pmf calculations between residue pairs
      • scores.txt - contains results for a crude scoring combination of contact probability and pmf, used in an earlier version of ZAM
      • distmethod.txt - identifies the method used for residue distances (CAs, CBs, etc)
      • analstats.txt - records parameters used during analysis (i.e., target temperature, frames read, etc)
      • chainpmf.txt - pmf calculations between chain centroids
      • umbrcontactprob.txt - contact probabilities for umbrella sampled residues
      • umbrpmf.txt - pmf calculations for umbrella sampled residues
      • *dist.txt.gz - distances for all monitored contacts for each trajectory frame; DELETED after analysis
      • mesoresults.txt - results from a mesostring analysis
      • clustresults.txt - results from a clustering analysis
      • clustXXX_YYpc.pdb - conformations from the clustering analysis (pc stands for percent)
      • clustmetrics.txt - contains various metrics calculated for each cluster, like radius of gyration, number of salt bridge pairs, etc
      • bestn*.txt - provides the best combinations of contacts as potential restraints, from punch analysis
      • metricresults.txt - contains various metrics calculated for each replica, like radius of gyration, number of salt bridge pairs, etc
      • eneresults.txt - contains average component energies calculated for each replica
      • punch/ - results from the punch stability and cooperativity analyses
    • conf/ - conformation directory.
      • seed*.pdb - user-supplied initial conformations
      • ref*.pdb - user-supplied reference conformations (for reference or anchoring restraints)
      • initXX.pdb - initial configurations prepared for amber where XX is a two-digit number starting at 01
      • refinitXX.pdb - reference configurations prepared for amber where XX is a two-digit number starting at 01
      • seedlog.txt - file describing alignments found during initial conformation generation
      • init-alignments.txt - file containing the final scores generated from the Monte Carlo run for assembly
      • snapXX.pdb - representative snapshot configurations where XX is a two-digit number starting at 01; these are identical to the cluster conformations in anal/ unless options are specified to remove redundant secondary structures or unstructured loops
      • rese*.pdb - user-supplied reservoir conformations
      • reselog.pdb - file summarizing reservoir files found, and their energies
      • rese.dat - a library of reservoir structures generated from supplied reservoir pdbs (allows fast swapping of reservoir structures)
    • data/ - REMD data directory
      • rexstats.txt - records parameters for the REMD simulation (e.g., num of replicas)
      • seq.txt - records the sequence used in the REMD simulation (incl. caps)
      • temps.txt - list of temperatures used in the REMD simulation
      • chains.txt - describes the residue ranges of different chains found by ZAM
      • states.txt - gives the temperature number and umbrella number of each replica
      • replicanum.txt.gz - list of replica numbers at each state, at each cycle
      • statenum.txt.gz - list of state numbers for each replica, at each cycle
      • restraints.dat.gz - data containing information about different restraints in replicas, used by WHAM
      • X.current.pdb - latest configuration from replica state X
      • X.mdene.txt.gz - energy history for replica state X
      • X.mdtrj.crd.gz - accumulated trajectory for replica state X
      • X.prmtop.parm7 - parameter and topology file for replica state X
    • work/ - REMD working directory
      • restart.dat - restart data for the REMD simulation run
      • X/ - directory for replica X containing all AMBER files

Customization[edit]

Initial structures[edit]

The user can supply initial structures for a ZAM fragment in the form of pdb files. If the fragment has no parents, these structures are used instead of the extended structure for the initial conformation. If the fragment has, these structures are used in addition to the parent structures; in this case, the user-supplied structures have higher priority than the parent ones and will be used in lower temperature replicas.

To supply a fragment with structures, simply place files named seed*.pdb in the conf/ subdirectory of the fragment prior to it being run. When run, ZAM will first sort the files alphabetically, giving priority to those which come first in the sorted sequence. It will then check the sequences of the seed files and make alignments to the fragment sequence; mismatches in sequence will cause ZAM to truncate the parts of the pdb file it uses. ZAM will then generate new pdb files called conf/init*.pdb which are processed versions of the seed files: missing residues are added as needed, extraneous residues are deleted, caps are added, and the structure is prepared for running in AMBER. ZAM attempts to remove possible steric overlaps during this process, but if they are found and cannot be fixed, it will omit those seeds from the run. The file conf/seedlog.txt details the results of this process.

Seed structures need to conform to each other in terms of pdb convention and also in terms of AMBER's naming convention for heavy atoms (hydrogen atoms are rebuilt, so these are not important). If not, errors probably will occur.

If there are less init*.pdb files than replicas, ZAM will re-cycle through them in assigning to replicas. If there are more, ZAM will only use those structures up to the number of replicas.

Notes[edit]

Sometimes it is helpful to add a little text describing each fragment in plain language. This is possible using the command zam.py fragnote. For example,

zam.py fragnote 5 This is a test of the 5-2 restraint

will add the text This is a test of the 5-2 restraint to the output of zam.py show for fragment 5. Not entering text after the fragment number is equivalent to clearing the note.

Caps[edit]

To explicitly override caps, one can use a command like

zam.py fragcaps 5 ncap

to specify only a n-terminal cap added to fragment 5. The final token can be none, both, ncap, or ccap. By default, all fragments are capped.

Number of replicas[edit]

By default, ZAM chooses a number of replicas appropriate for the size of the protein and the temperature range so that the acceptance ratios for swaps will be roughly around 50%. This can be overridden, however. This might be used, for instance, to specify more replicas than would be normally chosen so as to accomodate a larger set of initial structures, if a concensus type run were being performed. The command

zam.py fragnumtemps 5 60 ------Updated 2022/06/21
zam.py editfrag 5 NRep 60

will set the number of replicas for fragment 5 to be 60. Note that proper capitalization of NRep is important here. To reinstate the automatic determination, set the number of replicas to zero. Note that the number of replicas should only be changed before a fragment has been run, or after a zam.py reinitfrag command. Changing it after a fragment has started will result in major problems.

Replica exchange temperatures[edit]

The range of temperatures used for an individual fragment can be modified using the command fragtemps. For example, the command

zam.py fragtemps 5 270 400

sets the minimum and maximum temperatures of fragment 5 to 270 K and 400 K, respectively.

Other parameter changes[edit]

There are a large number of global parameters that can be changed in a ZAM run. These are listed in a file called zamsetup-template.py. This file provides all parameters, their default values, and a text note describing their effect. If it is desired to change a default parameter, the procedure is the following: one creates a separate zamsetup.py file and places it in the ZAM project folder (not the ZAM installation folder). Then, only those changes from default that are desired are entered into zamsetup.py from the template version. It is important that if any parameter is a variable in a sub-module (such as zamrex.MinTemp) that there be a corresponding import statement for the module at the beginning of the file (such as import zamrex). During a run, ZAM scans for the presence of this file and uses it to make modifications to default variables.

Adding restraints[edit]

Residue-residue[edit]

This is the most common kind of restraint to add to a fragment run. By default, the restraint is added between residue centroids with a force constant of 0.5 kcal/mol/A^2. When adding restraints, residue numbers remain relative to the master sequence, and not to the particular fragment sequence (i.e., 8 always means residue 8 in the master sequence, not in the fragment sequence.) Here are some examples of adding restraints:

zam.py addrest 5 4 7
adds a restraint between residues 4 and 7 in fragment 5.
zam.py addrest 5 4 7 5 8
adds restraints between residues 4 and 7 and 5 and 8 in fragment 5.
zam.py addrest 1-3,5 4 7 5 8
adds restraints between residues 4 and 7 and 5 and 8 in fragments 1, 2, 3, and 5.
zam.py delrest 5 4 7
deletes restraint 4-7 from fragment 5.
zam.py clearrest 5
clears all restraints in fragment 5.
zam.py addglobalrest 4 7
will add the 4-7 restraint to the global list of restraints, which will then automatically be applied to any fragments created after that point (though they can later be edited on a fragment by fragment basis).
zam.py clearglobalrest
will clear all the global restraints
zam.py delglobalrest 4 7
deletes the global restraint 4-7.
zam.py copyrest 5-7 2
copies all restraints from fragment 2 to fragments 5, 6, and 7.

To show the restraints in a fragment, use the command

zam.py showrest .

The added restraints are called "Prior restraints" because they are added prior to a simulation run. Other kinds of restraints are "Proposed restraints", which are suggested by ZAM after the analysis run (but not used during the run), and "Swap restraints", which are restraints that are different in each replica and swap with the configuration (an experimental method).

Reference torsional restraints[edit]

It may be desirable to hold part of a structure in a particular conformation, in all replicas. To accomplish this, add backbone torsional restraints that hold specific phi/psi angles in the same conformation as that of a reference file. The first step in doing this is to supply a reference file. This will be automatically taken as the first seed conformation (when sorted alphabetically), but it can be overridden by supplying conf/ref.pdb. Actually, more than one file can be supplied, ala ref*.pdb; these will be distributed across replicas in the same way that initial conformations are, after being sorted alphabetically. Then, to specify which residues are to be held in the reference conformation use the zam.py rigidifyfrag command, such as

zam.py rigidifyfrag 5 1-5,20-25
to hold residues 1-5 and 20-25 in the reference phi/psi angles for fragment 5.
zam.py unrigidifyfrag 5
to remove these restraints

It is important to note that seed conformations for such runs already need to be close to the reference conformation; otherwise, instabilities can occur during the initial phases of replica exchange due to high energies in the violated restraints.

Secondary structure restraints[edit]

ZAM offers the ability to automatically add phi/psi torsional restraints that hold the backbone in specific secondary structures:

zam.py addss 5 ---HHHH--EEEE-
will enforce a combination of helical and extended secondary structure in fragment 5. Currently, H and E are the only available options. Note that the secondary structure string must be the same length as the fragment.
zam.py clearss 5
removes secondary structure restraints from fragment 5.
zam.py addglobalss ----HHHHH---EEEEEE----HHHH
will enforce a global list of secondary structure restraints, and any fragment created after this command will automatically inherit the restraints (they can, however, be overridden).
zam.py clearglobalss
to remove any global secondary structure restraints.

When secondary structure restraints are added to fragments without parents, ZAM will perform a special configuration-making process, similar to assembly: it will automatically generate the desired secondary structures (rather than using a completely extended structure), and it will optimize packing using the rigid body Monte Carlo program. On the other hand, if a fragment does have parent structures, ZAM will use them as per the regular protocol, depending on whether the stage is grow or assem. Again, in these cases, it is important that any initial seed conformations or parent structures already have the correct secondary structures in place. Failure to do so can cause the simulation to crash due to high restraint energies. This means that, if new residues are to be added to an existing fragment, and those new residues have helical secondary structure restraints (for example), assembly should be performed to add these new residues from an existing helical conformation, rather than adding them in extended form.

Anchoring restraints[edit]

Another method to hold a part of a structure in a reference conformation is to use anchoring restraints, essentially harmonic springs that tie either backbone atoms or all atoms down to a point in Cartesian space. These restraints offer greater control in enforcing a particular conformation, and are also useful for anchoring one chain in place at all temperatures while another chain samples relative orientations, as in docking. Similar to the case of reference torsional restraints, a seed pdb file must be supplied, which serves as a source of the reference structure. Then, to add anchoring restraints use:

zam.py anchorfrag 5 1-10,20-30 11-19
will add backbone anchoring restraints to residues 1-10 and 20-30, and all-atom anchoring restraints to residues 11-19
zam.py anchorfrag 5 all 11,12,15
will add backbone anchoring restraints to all residues, with additional all-atom restraints to residues 11, 12, and 15
zam.py anchorfrag 5 1-10 none
will add only backbone anchoring restraints to residues 1-10
zam.py unanchorfrag 5
will remove backbone anchoring restraints

Note that multiple calls to anchorfrag are not additive. That is, a second anchorfrag command overrides the previous one. Also, it is important that all seeds are close to the same reference structure where the anchored residues are concerned, for stability of the simulation.

Advanced topics[edit]

Reservoir swapping[edit]

ZAM can allow "reservoir swapping" into one or more replicas during replica exchange. By default, this is the last replica (highest temperature), the reservoir temperature is infinite, and swapping occurs before the MD trajectory of each REMD cycle. To enable reservoir swapping, simply place structure files of the format rese*.pdb in the conf subdirectory of the relevant fragment and these files will automatically be recognized and processed as reservoir structures. A log file reselog.txt will be generated in that folder listing the structures found. Progress files for the fragment will then reflect the reservoir swaps. Reservoir files must be present during initial configuration making, and the addition or deletion of files after that has taken place will have no effect. Note that reservoir swapping formally breaks detailed balance in the replica exchange simulation.

The following are options that can be added to a zamsetup.py file in the main folder: 

import zamrex, zamrun
#number of reservoir swaps to attempt each cycle
zamrex.ReseNSwap = 1
#temperature of the reservoir
zamrex.ReseTemp = 1.e200
#whether to energy-minimize reservoir structures
zamrun.MinResePdbs = False

Salt bridge repulsion[edit]

By default, ZAM adds restraints to every potential salt-bridge atom pair, centered on the heavy atoms. This includes charged termini atoms. The restraints are replusive, designed to prevent salt bridges from forming. The function can be turned on and off on a fragment by fragment basis:

zam.py repelsaltbridges 5 off 
turns the repulsion off in fragment 5
zam.py repelsaltbridges 5 on 
turns the repulsion back on

Multiple chains[edit]

It is fairly straightforward to start a simulation with multiple chains, such as a docking simulation. To do this, one must explicitly specify the terminal residues in the initial master sequence. N- and C-terminal residues begin with the letters N and C, respectively and per AMBER's convention, such as NALA, CLYS, NTRP. Capping residues can also be explicitly specified: ACE, NME, and NHE. Here is an example that starts a simulation with two dialanine chains, where the second is capped:

zam.py initializerun NALA CALA ACE ALA ALA NME

Internally, ZAM relies entirely on AMBER's processing of the sequence to determine how many chains there are. A good way to test the effects of a particular sequence, if having problems, is to use the tleap command sys = sequence{...} and produce a pdb file to check that multiple chains were indeed achieved.

There are some important caveats with multiple chains: (1) currently, fragments must be contiguous sections of the master sequence; therefore it is impossible to have the first chain be complete and the second chain a small interior subsection, without setting up a separate ZAM simulation; and (2) ZAM will not cap a terminal residue or another cap, even if the command fragcaps is used.

In order to prevent the chains from becoming vastly separated in space at higher replica temperatures, ZAM automatically adds a restraint between all chain pairs. The restraint is determined as follows: the radius of gyration is calculated for each chain based on the initial structure. This value is scaled up by a factor of (5/3)^(1/2) to approximate the radius of the surface of each chain (in the spherical approximation). The alpha carbons closest to the centroid of each chain are selected and a restraint is added between each pair. The restraint is zero until a distance equal to the sum of the chain surface radii plus an additional amount (default is 10 A), at which point it becomes harmonic with a force constant of 1.0 kcal/mol/A^2. One note: when umbrella sampling is enabled between residues in two chains (see below), the corresponding chain restriant is skipped.

For docking a ligand to a protein, it is often useful to anchor the protein backbone positions to Cartesian space, so that it does not unfold at higher temeratures. Moreover, it is important that seed conformations are given with the exact fragment sequence, as ZAM's alignment protocol is not capable of reconstructing missing residues or deleting extraneous ones in a way that preserves good conformations when more than one chain is present.

Umbrella sampling[edit]

ZAM permits umbrella sampling along the distance between two residues or the centroid distance between two chains. Currently, if umbrella sampling is desired, ZAM will extend the replica space to two dimensions, one in temperature and one in distance. There will be two temperature chains, spanning from the minimum to maximum temperature, at each of the distance extremes. Connecting them will be a single distance chain of replicas, only at the lowest temperature. Swaps are permitted between any two replicas that are separated by at most one temperature and one distance difference.

To set up an umbrella sampling simulation, use the following commands:

zam.py addumbrellarest 5 2 10 
to add umbrella sampling between residues 2 and 10 in fragment 5
zam.py addumbrellarest 5 -1 -2 
to add umbrella sampling between chains 1 and 2 in fragment 5
zam.py clearumbrellarest 5 
to stop umbrella sampling in fragment 5
zam.py umbrelladistances 5 10. 15. 20. 25. 30. 
to specify the umbrellas at distances 10, 15, 20, 25, and 30 A in fragment 5
zam.py umbrelladistances 5 span 10. 30. 5. 
to specify the umbrellas at distances starting from 10. A and going to 30. A in 5. A increments

During an umbrella sampling run, ZAM will add a harmonic restraint in each umbrella to limit sampling to a particular distance, depending on the umbrella index. The restraint will be centered around the target distance and will have a flat region where its value is zero (by default, the width of this region is 5 A). Only in the first distance, there will be no left-hand side to this parabola (i.e., no restraint at short distances). The replica exchange swap probabilities and WHAM reweighting methods will automatically account for differences in restraints among the replicas. After the simulation is run, the files umbrpmf.txt will be produced in the analysis folder, to show the PMFs generated along the umbrella contacts.

Swapping restraints[edit]

An experimental option, ZAM can add special restraints that differ in replicas and follow configurations after swapping. This can be a useful way to track the most favorable contacts by watching which restraints swap to lower temperatures in replica exchange. However, this protocol violates detailed balance, and is best used with caution. When swapping of restraints is enabled, ZAM no longer accounts for restraint energies in acceptance probabilities or in WHAM, since it is unclear how to do so. To edit swapped restraints, use:

zam.py addswaprest 5 1000. 2 5 3 8 
will distribute the restraints 2-5 and 3-8 across the replicas in fragment 5 and allow them to sawp, only for the first 1000 ps, after which the swapped restraints will be removed

Scaling restraints[edit]

ZAM can scale the strength of restraints from 0 to 100% strength as one moves from the lowest to highest temperature replica. In that way, the lowest temperature remains restraint-free for the purposes of clustering, reweighting, free energies, etc. It is possible to do this in a way that retains detailed balance, and ZAM automatically makes the necessary adjustments to the replica exchange swap probabilities and the WHAM reweighting protocol. The scaling applies to residue contact, anchoring, secondary structure, and reference restraints. The salt bridge repulsion is not scaled, however. To enable such scaling, use:

zam.py scalerest 5 on 
to turn on restraint scaling in fragment 5
zam.py scalerest 5 off 
to turn it off

Note that scaling restraints may result in low acceptance probabilities for swaps between replicas, if the strength of the restraints is high or there are too few replicas. The strength of restraints can be modified using a zamsetup.py file, and the number of replicas can be specified using zam.py.

Disulfide bonds[edit]

Disulfide bonds can be created from the command line. To set a disulfide in the overall sequence for a ZAM run, enter: 

zam.py addglobaldisulfide RESNUM1 RESNUM2

where RESNUM1 and RESNUM2 are the residue numbers of the two CYS residues between which the disulfide will be made.

Note that, when ZAM runs a fragment containing a disulfide bond, it first checks any initial structures to make sure that the disulfide atoms are close enough to each other for the bond to form. If these atoms are not close enough, ZAM will modify the initial structures so that the simulations will be stable. These modified structures are reflected in the init*.pdb files.

IMPORTANT NOTE: Global disulfides must be added BEFORE a fragment is created. To add disulfides to existing fragments or to specific fragments (not all), use: 

zam.py adddisulfide FRAGNUM RESNUM1 RESNUM2

Other amino acids[edit]

If other amino acids are to be used, several modifications need to be made to the ZAM. Take the example of D-lysine, which we will use lys (in lowercase) to designate. First, a tleap library containing the new residue needs to be created manually for AMBER. For D-amino acids, one way to do this is the following (in tleap): 

source leaprc.ff96
lys = copy LYS
set lys name "lys"
set lys.1 name "lys"
transform lys {{1 0 0}{0 1 0}{0 0 -1}}
check lys
saveoff lys lys.off
savepdb lys lys.pdb
quit

The next step is telling ZAM about the newly created residue. A zamsetup.py file will need to be created. Then, there will be three steps: (1) specifying build strings for tleap in AMBER that import the new moities, (2) adding the new residue information to ZAM's sequence library, and (3) adding structures of the new residues to ZAM's structure library. Here is what that looks like in zamsetup.py: 

#step 1 -- load library into amber when using tleap
zd.PreBuildString = "loadoff lys.off"
#step 2 -- add to sequence library in ZAM
import sequence
sequence.AminoAcids.append(
  sequence.AminoAcidClass(Name = "D lysine", AA1 = "k", AA3 = "lys", 
    Hydrophobic = False, Charge = 1, Natural = False, Cap = False,
    AA3Aliases = ["nlys", "clys"], SaltAtoms = ["NZ"]))
#step 3 -- add to structure library in ZAM
import protein
protein.AddTemplates("lys.pdb")

If your residue is a cap, you need to use the following instead: 

#step 1 -- load library into amber when using tleap
zd.PreBuildString = "loadoff NCAP.off"
#step 2 -- add to sequence library in ZAM
import sequence
sequence.AminoAcids.append(
  sequence.AminoAcidClass(Name = "N-cap", AA1 = "X", AA3 = "NCAP", 
    Hydrophobic = False, Charge = 0, Natural = False, Cap = True)
#step 3 -- add to structure library in ZAM
import protein
protein.AddTemplates("NCAP.pdb")
protein.NoDihedrals.append("NCAP")

All of these changes should be made *before* the ZAM run is initialized (zam.py initializerun). Once these changes are made, however, it should be possible to use the residue lys like any other residue in ZAM. Recall zamsetup.py and associated .off and .pdb files should be in your zam project directory, not the installation directory for ZAM.

One caveat about nonstandard amino acids: you should try to keep the normal pdb atom names as much as possible in the pdb files you use. In particular, ZAM uses the atoms "N", "C", "CA", "O", "CB", "H", and "HA" to make decisions about how to connect up peptide bonds, etc.

Cyclic peptide[edit]

To build a cyclic peptide, one needs to (1) remove charged termini and (2) connect two ends by adding a peptide bond. (1) can be done by add zamsetup.py with the following codes. Then, a peptide bond can be added using "zam addbond FRAGNUM RESNUM1 ATOMNAME1 RESNUM2 ATOMNAME2" 

import mdsim
add mdsim.ChargedTermini = False


Changes to AMBER MD settings[edit]

All settings used in the AMBER suite (tleap, sander) by ZAM are contained in string templates in the file mdsim.py. Many settings, such as the force field, implicit solvent model, and salt concentration, can currently be adjusted using an appropriate zam.py command. This is the preferred method, though string templates may also be adjusted, as shown below, for other parameters. For instance, to set the salt concentration to 0.4 and the force field to ff12SB with igb8 in fragment 0, one would use: 

zam.py saltconcentration 0 0.4
zam.py forcefield 0 ff12SB_igb8

If it is desired to tweak other run parameters, such as the surface area calculation method, the cutoff, etc, these changes can be inserted into a zamsetup.py file. To do so, locate the variable in mdsim.py that contains the relevant template. The variable TLeapDflts contains default parameters for tleap, such as the force field and PB radii used. The variables RunTmplMin and RunTmplMD contain the default input for sander minimization and molecular dynamics runs, respectively. Once the relevant variable is found, copy its definition in mdsim.py to a zamsetup.py file in the ZAM run directory. Be sure to add import mdsim and the mdsim prefix (ala mdsim.variable) to the text in zamsetup.py. Then, make modifications to the definition as necessary. Here is an example zamsetup.py file that changes the cutoff to 20.0 (ZAM default is 16.0): 

import mdsim
mdsim.RunDflts = { "STEPSMD":500, "STEPSMINTOT":250, "STEPSMINSD":200, "STEPSMINCG":50,
            "STEPSSAVE":500, "STEPSREMOVECOM":500,
            "VELOCITYLIMIT":0.,
            "TEMPSET":270.0, "TEMPSET1":270., "TEMPSET2":270., "STEPSTEMP":500,
            "TEMPMODE":0, "BERENDSENTAU":1.0, "LANGEVINGAMMA":0.0,
            "SEED":314159, "STEPSIZE":0.002,
            "WEIGHTSOPT":"", "STEPSWEIGHT1":0, "STEPSWEIGHT2":500,
            "RESTSCALE1":1.0, "RESTSCALE2":1.0,
            "NONRESTSCALE1":1.0, "NONRESTSCALE2":1.0,
            "RADIUSSCALE1":1.0, "RADIUSSCALE2":1.0,
            "ELECSCALE1":1.0, "ELECSCALE2":1.0,
            "NONBONDSCALE1":1.0, "NONBONDSCALE2":1.0,
            "POSRESTON":0, "POSRESTOPT":"", "POSRESTFCONST":0.01,
            "SHAKEMODE":2, "SURFACETENSION":0.005,
            "INPUTMODE":1, "RESTARTVEL":0,
            "IGB":5, "GBSA":1, "CUTOFF":20.0,
            "SALTCON":0.0,
            "MODEL":"ff96glghs"}

Automated running[edit]

For protein folding simulations, ZAM is capable of semi-automated running. Once a sequence is entered using the initializerun command, ZAM can make decisions about fragments to grow/assemble and restraints to use on its own. To let ZAM decide what to do on its own, use the command

zam.py decidenext .

During later stages of a simulation, this command can require some significant time to run, because it will consider many fragment combinations as possible routes to pursue, so it is advised to run decidenext as a job and not at terminal.

Currenly decidenext follows a procedure consisting of four stages:

  1. 8mers are made throughout the entire chain and simulated, without restraints
  2. 12mers are grown from the top 2/3 most structured 8mers, without restraints
  3. 16mers are grown from the top 3/4 most structured 12mers, without restraints
  4. Various combinations of the generated fragments are assembled together, picking restraints from the parent runs; increments of 10 assemblies at a time is the default

Running decidenext will setup the next stage on the list. If stage 4 has been reached, it will be repeated. Each time, this command will create new fragments to be simulated. Then, the user is required to submit the simulation to run the created fragments. Afterwards, a subsequent call to decidenext will make further decisions.

Regardless of whether or not decidenext is used, ZAM will publish information about residue-residue contact probabilities that it continously estimates. These can be found in the file logit.txt and visualized using logit.html. For each contact pair, the logit is the log ratio of the probability a contact is native to nonnative. This ratio is estimated using measured data from each fragment that contains both residues in the contact.
