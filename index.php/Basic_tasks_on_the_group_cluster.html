How to log onto the group cluster[edit]

Our group cluster is housed remotely in CNSI and currently has 144 available CPUs and 8 GPUs. To log onto it from Windows, use either Putty (for terminal access) or WinSCP (for transferring files). From linux or OsX machines, you can simply use a terminal and ssh. The login server is zin.cnsi.ucsb.edu, so you will want to log in to <yourusername>@zin.cnsi.ucsb.edu.

Below are some helpful hints for configuring Putty if you are using it:

  • You should enable “keepalives” which will prevent your session from being closed during idle periods. To do this, in the Putty configuration screen, go to the “Connection” category and enter a number like 10 in the field “seconds between keepalives”.
  • OPTIONAL: If you have installed Cygwin and its X11 packages (Note: There is a local mirror of Cygwin), it is possible to interact with the remote graphical windows directly on your desktop. First, start the xwin server on your workstation by running c:\cygwin\usr\x11r6\bin\startxwin.bat. You can close the terminal that appears, but an “X” should now appear in your system tray indicating the xwin server is running. Then, go into Putty and to the configuration category “Connection > SSH > X11”. Check the box for “Enable X11 forwarding” and enter in “display location” the value “localhost:0”. Then connect to the group cluster per usual. Now, if you run applications with windows (e.g., firefox), then these will appear in full on your workstation screen.

Once you log into the cluster, your home directory should already be preconfigured. Common group programs and scripts are kept in the location /share/apps. Please do not modify anything in the share directory unless there is a group decision that you should do so. Your path variable should also be preconfigured so that you can directly access these without having to reference the path. You may also load additional software that using the module system, as described below.

Installing new software on the cluster[edit]

When you have to install new software for your own use on the cluster, its a good idea to first figure out if multiple members will need it or if you are the lone ranger. If its the latter, consider doing a local install in your home directory. Additionally, it is not recommended to run GUIs on the cluster. If you need to install / use something that renders graphic visualizations such as VMD or PyMol, try to see if there are text only versions of that software (e.g. the VMD version that is currently installed is text only without GUI support). Remember, the cluster is for running batch processes, and not fancy visualizations which should be typically be done off your personal workstations. Whenever you install new software, make a note of the steps that you followed. If possible talk to your sysadmin to include this information as a separate file in the system directory /share/apps/installation_scripts (more details given later)

Using environment modules[edit]

The cluster is set-up to use a "module" system to manage and load installed software. If you want to run some software and cannot seem to get it to run, please see if the requisite module files are loaded. E.g. different versions of LAMMPS may be required for different people. So, by default NO LAMMPS modules are loaded, and you’d have to load your own. Same is true for a bunch of other software too.

You can see which packages are available using the module avail command, and those that are currently loaded with module list. To load a module from the list of available modules type module load <modulename/version>, and correspondingly to unload a module that is no longer required use module unload <modulename/version> . That should correctly set up all environment variables, including paths, that are necessary for the code to run. Note that most of the modules available through this system are things that require the code to be compiled manually, which generally takes a lot of time to do right. To add a module-file, you will need root access. This is thus considered an administrative task and is described in the Cluster setup part of the wiki. Alternatively, many packages can be installed and managed locally (i.e. on a user rather than system level) through things like anaconda or pip (as described below).

Different python environments on the cluster[edit]

Note that the default python version is the "vendor supplied" version that came with the Rocks 6.2 OS. This is a super-old version that you should not use, but should also NEVER delete. The default python that is loaded for you at login is the python-2.7.13 version that uses the "pip" package manager. If you wish to use this as your standard python, then using pip install <packagename> --user will install the required package under the ~/.local folder in your home directory. Alternatively, if you wish to use the more feature-rich anaconda version, then you can make it your own go-to default by putting the line module load python/anaconda in your .bashrc file.

Using anaconda as a python package manager[edit]

The anaconda package manager is a useful tool for installing and managing many python packages useful in general computing and molecular simulation. You can easily install anaconda by downloading the appropriate installer (for the cluster, this will be linux) and then running it from the command line with bash Anaconda-version-Linux-x86_64.sh. You will be prompted to change the install location if you want and at this point you will want to make sure it is installing to a location that you own (i.e. is inside of your home directory somewhere). To install other packages via anaconda, it's as easy as typing something like conda install pacakagename. All of the libraries and other packages that the package depends on will automatically be installed, avoiding a lot of headaches. It comes with the latest stable versions of numpy and scipy already installed, but you can also install things like scikit-learn or ParmEd. If something gets messed up and you want to uninstall and start from scratch, all you have to do is remove the directory where it was installed (and maybe modify your .bashrc file to stop trying to access this directory). This makes it an easy and flexible way of setting up your local environment to run your code on any cluster that you happen to be using.

Names of executables / libraries on the cluster that can be loaded as modules[edit]

  • LAMMPS: "lmp_mpich2" (points to whichever LAMMPS version (15May2015 or 31Mar2017) is loaded as a module)
  • GROMACS: (the usual) "grompp" for compiling and "mdrun" for running simulations. Also, topology manipulators like "trjconv", "trjcat", "editconf", etc. Can point to one of any of the gromacs/x modules where x = 4.6.5, 4.6.5-mpich2, 5.1.4, 5.1.4-mpich2, 2016.1, 2016.1-mpich2, 2018.2. mpich2 suffixed versions enable distributed memory parallelization.
  • AMBER: load the amber/amber16 module. More details about GPU enabled amber can be found in the page on Syrah.
  • Pymbar: Use import pymbar in python scripts.
  • WHAM: Two flavors are loaded. One is the C++version from the Grossfield lab which can be accessed through the binary wham once the wham/grossfield module is loaded. The other is a fortran library written by M. Scott Shell; to use this load the wham/mss module and then use import wham in python scripts.
  • parallel runs: "mpirun" or "mpiexec". These are enabled through the mpi/rocks-mpich2 module that is loaded automatically loaded at login. DO NOT UNLOAD this module.
  • VMD: "vmd" (points to the NON-GUI (text only) version once the vmd/1.9.3_textmode module is loaded)
  • PyMol: "pymol" (points to the open-source PyMol version once the pymol/1.8.6 module is loaded. Note, this is not the proprietary version distributed by Schrodinger)
  • CMAKE: "cmake" (automatically loaded at login; DO NOT UNLOAD)
  • PACKMOL: "packmol" (points to the packmol module). Useful for building atomistic topologies.
  • STRIDE: "stride" (points to the stride module). Useful for secondary structure calculations for atomistic peptides.
  • GNU compilers: Latest GNU compilers for C, C++ and fortran are provided through the gcc/gcc-7.1.0 module which is loaded at login. DO NOT UNLOAD this module, unless you want to break everything.
  • Python: Read the discussion on different python modules above. In any case, do not use the default python2.6 version or the one pointed to by the python/rocks module. These are ancient.

Checking your space and resource usage[edit]

From your home directory (cd ~), the following command will show how much space you are using: 

du -ch

You can also check the space of all drives on the cluster: 

df -h

To check how many compute cores are free for job submission:

freecores

Data compression[edit]

Please try and use compressed formats for storing molecular dynamics trajectories and thermodynamic log files where possible. E.g, GROMACS will already output trajectory data in a compressed .xtc format. But, others, especially LAMMPS just outputs plain ASCII text files that can quickly take up a very large amount of cluster space. ALWAYS store LAMMPS trajectories in .gz or .bz2 format. LAMMPS can automatically write out .gz compressed files if you ask it to, as described here. If you need to unzip an trajectory to visualize in VMD, PyMol, etc, copy it to your local workstation. For more information on these compression formats, look here.

You can look into python's gzip and bz2 modules for directly reading compressed files inside a python script. For large LAMMPS trajectories, you can use the pickleTraj module to read the trajectory once using the sim package into a pickle. When you load the trajectory the next time from the pickle, access is very fast. Usage: 

import pickleTraj
trj = pickleTraj(<lammps_trajfile_name>)

Note however, that python "pickles" (and other object serialization formats like python "shelves") can sometimes take up a lot of memory. Pickling trajectories make sense because the lightning fast accession speed outweighs the memory cost. But the same trade-off might not hold for other simpler data generated from post-processing and it might make more sense to simply dump such data into an ASCII file (.txt or .dat) and then compress that as discussed above.

How to submit simulation jobs on the cluster[edit]

The cluster has a finite number of processor resources. To manage simulations that are run on it, we employ the Sun Grid Engine (SGE) software, which is fairly straightforward to use.

Here is how SGE works: one submits a job to be run to SGE, which places it in a job queue. The job is actually a shell script that contains a header with some pertinent information about the job (e.g., its name) followed by the commands you want to run. SGE constantly monitors the number of available (unused) processors in the cluster. When one becomes available, it picks the next job in the queue, assigns it to that processor, and starts running it. It will never start more jobs than there are available processors.

The following describes some common tasks:

Make a new job script[edit]

A template for job scripts is found at /share/apps/job_serial.template for general simulations (note that this only runs jobs using a single core, not those using multiple cores - use job_parallel.template for parallel jobs using multiple cores). Copy this file to the folder in your home directory from where you will run the simulation (you may want to rename the file suffix to ".sh" as it is really a bash script). Edit the file to put a job name in it and whatever commands you want to run, by replacing dummy lines present in the template. Be sure not to remove any other options, as this may cause your job to fail.

A note about email notifications: with the new system, email notifications have not been tested and may not be available. If you want to be notified when your job starts/finishes, this is theoretically possible with the SGE queuing system, but may need to be properly set-up.

Submit a job to the queue[edit]

Just type qsub job.sh where job.sh is your job script file and the job will be added to the queue.

To best allocate resources on the cluster, however, everyone is encourage to submit jobs using "smartsubmit", unless you really know what you're doing. For information on using "smartsubmit", just enter this into a console.

Note 1: If your submission consistently has the status message under qstat of "Eqw", contact the sysadmin about updating the user database on the nodes.

Note 2: To prevent wasting resource and time, it is always a good idea to test that all the parts of your code works by submitting a small / dummy version of your job, e.g. by using a low number of MD iterations, etc.

Note 3: If using smartsubmit, avoid using dashes in the name of the job submission script.

Submitting a parallel job[edit]

The valid mpi distribution on the cluster is mpich2. The cluster has another mpi version called openmpi that was "vendor-supplied" i.e. came with the OS. This version is BROKEN. So do. not use it! The mpich2 version gets automatically loaded for you as a module at login. DO NOT unload this. As mentioned before, a sample parallel jobscript job_parallel.template is kept in /share/apps. Notice that this sample script specifies 8 computational cores. The cluster is divided into separate "nodes" which all have 8 cores per node. So, if and when you need more than 8 cores, it is recommended that you use core numbers that are multiples of 8, for optimal speed. The smartsubmit script can take care of the rest.

List jobs currently running and in the queue[edit]

Type “qs” to see a list of all jobs. (Alternatively, for a more detailed display, type "qstat".) The “r” tag means a job is running and the “q” tag means it is waiting in the queue. By default "qstat" only shows jobs that you have submitted. To see all jobs in the queue, from all users, type "qstat -u '*'". You can also the shortcut command "mq" to view the folder locations from where your jobs are running.

Kill a currently running job or a job in the queue[edit]

Type “qdel JOBNAME” or “qdel JOBNUM” where you replace JOBNAME by the one you provided in the job script file, or JOBNUM by the number that is associated with the job in a qs/qstat output.

Submitting / killing multiple jobs at once[edit]

ZAM simulations require multiple submissions of the same job. A utility called “multijob.py” may be found in /share/apps/scripts and allows you to do this all at once. To submit job.sh 11 times, you can call “multijob.py sub 11 job.sh”. To kill all jobs with a specific name, you can type either “qdel JOBNAME” or “multijob.py deljob JOBNAME”. To kill all jobs with a range of job numbers from X to Y (inclusive), type “multijob.py delnums X Y”. Note that when your jobs start running, an output file will be created with the name “JOBNAME.oJOBNUM”. This file captures whatever output your program produces and can be a good place to examine possible bugs in your code if your job stops prematurely.

CAUTION: "multijob.py" is relatively old code and may not behave properly in all situations. It is better to use "smartsubmit" to submit multiple jobs, as this helps keep more full nodes free for parallel jobs. For killing or otherwise managing jobs, an appropriate command-line tool like "qdel" is likely all you need. For instance, a range of jobs may easily be killed with "qdel {startnumber..endnumber}". Many other such tricks may be found using google.

NFS Structure[edit]

The major "system directory" on the cluster is /share/apps. This is a Network File System (NFS) mount, which means that while the actual data is physically stored elsewhere on the RAID array, /share/apps provides a hard link to this data that is automatically shared across all the cluster nodes. The following describes a rough hierarchy of subfolders within this directory that may be necessary for users. (Note that all content here is read-only, you need either root access or help from your sysadmin to create/edit any files here):

scripts[edit]

This contains all group code (mostly python and fortran scripts). Python scripts that make up the zam and sim packages are kept here. All of the scripts / code here is automatically added to PATH and $PYTHONPATH environment variables, so that you can run / import any sofware kept here automatically.

installation_scripts[edit]

Here you'll find a number of shell scripts / instructions that lists all the dependencies that were required for installing a particular software and how to install each individual dependency. If you find complete shell scripts for installing a particular software, NEVER run it as is. The scripts here are intended to provide step-by-step instructions and not plug-and-play solutions to get a particular software. Also, whenever you install new software, try to compile a list of instructions and (if this is an important software which more lab members are going to use), ask your sysadmin to put it in a file under this folder. You can dump any additional things you want into that file or folder. Files under this directory should be named as get_<software>.sh. Update the README if required. If a software has plugins built by the group that need to be put in at compile them, you’ll find them here.

admin_scripts[edit]

Treat all code in here with EXTREME CAUTION. Blindly executing any of these shell scripts can instantly break yours or others' work. The scripts here are useful for propagating changes to compute nodes if they are rebooted or reinstalled. Contact your sysadmin or M. Scott Shell if you don’t understand something in here. Bottomline: Do not run anything from here as is without prior confirmation. (If you still have further questions, shoot an email to tanmoy.7989@gmail.com)

Mounting your cluster home directory on a local workstation[edit]

As mentioned before, it is not a good idea to run any kind of graphics on the cluster, since that makes I/O slow for everyone else. The only applications that you may run safely without slowdown are linux's simplistic text editor gedit and figure viewer eog. To access cluster data and create visualizations with say PyMol or VMD, you can mount your home directory to a local machine (your laptop or office workstation). To do this, run sshfs <yourusername>@zin.cnsi.ucsb.edu:/home/cask0/home/<yourhomedirname> <local location> from your local machine.

Backing up data (IMPORTANT)[edit]

It is good practice to backup data every few months from the cluster to a local archive (say a portable hard drive). This is especially useful when a project is complete and you don't need that data immediately, so that it can be removed from the cluster to free up space. The best way to do this is through rsync. You can find more details here Basically, all you need to do is run from the command-line of your local machine. However, make sure that not more than 1/2 persons are running rsync at a time, and if we're talking anything over ~2-5 GBs, best to have just one person running rsync at a time. Otherwise, you will put severe strain on the RAID array and may end up slowing down or completely compromising cluster I/O. Communicate with other lab members before archiving your work, and monitor the I/O usage on ganglia (look at the load on the "cellar-0-0" node)

Making your files readable by other group members[edit]

If you would like to give others in the group the ability to read files in your directory, navigate to a folder outside the folder you want to make readable and execute:

chmod -R g+Xr folder-name

-R is to recursively execute, g+ applies the permission to everyone in the group on top of whatever existing permissions there are, X gives execute permissions to folders (necessary to allow access to the folder), and r gives read permissions. If you execute this on your home directory, make sure that any files you want to keep secret (such as .ssh/id_rsa) are not given read permission.

For more information about chmod, execute:

man chmod
