System Information[edit]

Syrah is a single node with 8 GPUs and 2 CPUs, each with 8 hyperthreaded cores, allowing for the use of 32 logical "CPU" cores. The 8 GPUs are set-up so that peer-to-peer communication is only available between certain sets of GPUs. This can be checked with

/usr/local/bin/gpuP2PCheck

The output of this should show that GPU sets 0-3 and 4-7 can communicate efficiently. More GPUs can be used during runs, but at least for AMBER, this may actually slow down the calculations instead of speeding them up. Which GPUs are available to be used can be set with the environment variable CUDA_VISIBLE_DEVICES. The status of GPUs may be checked with the nvidia-smi command.

The system came pre-installed with CentOS 7 and a custom compiled version of AMBER16. A version of GROMACS (2016.1) compiled with GPU support specific to the machine has also been added. All AMBER environment variables should be pre-set for all users, however, to use GROMACS you will likely need to add the following to your .bashrc file.

source /usr/local/gromacs/bin/GMXRC

A very basic version of Python 2.7 came with the AMBER installation. A version with OpenMM installed is available by running:

export PATH=/usr/local/anaconda2/bin:$PATH

To upgrade or add Python functionality, locally install your desired version of Anaconda.

Adding and Removing Users[edit]

As root, run the following: 

adduser username
passwd username

And enter the desired password when prompted. The user will then be able to log in from the campus network using: 

ssh username@syrah.cnsi.ucsb.edu

Alternatively, the user may log in from Zin using ssh username@syrah or, alternatively, ssh syrah (assuming their Syrah username is the same as their Zin username).

To delete users, run: 

userdel usernmae

SGE Queuing System[edit]

The SGE queuing system has been set-up to monitor and allocate GPU resources. It has NOT been set up to allocate CPU resources. Instead, it is assumed that Syrah will be used for GPU simulations and that the CPUs should be left free to dynamically assist these GPU runs as necessary. However, because of this, be careful when submitting jobs with GROMACS, as its default behavior is to grab and use all of the resources it can see.

To monitor GPU usage, a load-sensor script /home/exx/sge_scripts/multi_gpu_sensor.sh is run by the SGE daemons every few seconds. This script utilizes output from nvidia-smi to decide how many GPUs are available for single-GPU jobs and multi-GPU jobs. This number is different because only certain sets of GPUs should be used together for multi-GPU jobs. The job submission template described below uses similar information to figure out which GPUs to send the job to. Sets of GPUs available for multi-GPU jobs with the largest number of running jobs are filled before empty sets. It is strongly recommended that the submission system be used along with the template submission script. To help prevent errors there is also a script called /home/exx/sge_scripts/jsv_gpu.pl that is run every time a job is submitted. This script checks for user errors in requesting GPU resources.

The queuing system may be modified with a GUI by typing qmon in a command line, or with a suite of command line tools. For more information on setting up a SGE queuing system (with a GPU load sensor), see the following resources: [1] [2] [3] [4]

For the queuing system to work smoothly, nvidia-smi must be able to quickly report on GPU resources. This is only possible if some program is running that constantly uses GPU resources (even just a little bit). If this is not the case, the GPU management and querying programs have to restart and get set up each time nvidia-smi is called, making it very slow and causing the queue to hang or break, requiring a manual restart. To avoid this, run (as root) after restarting:

nvidia-persistenced --persistence-mode

Running Jobs[edit]

A template for running GPU jobs with qsub on the SGE queuing system may be found in /home/exx/submit_example.sh. Note that the header of the file has most of the usual SGE variables defined, but also has: 

 #$ -l gpu=# desired gpus
 #$ -l multi_gpu=# desired gpus

This keeps track of the number of GPUs in use and allows the queuing system to run efficiently. To help the queuing system out, runs should be submitted 20-30 seconds apart to give time for the GPU job to actually start running. This lets the queuing system register that a certain GPU is in use and prevents all jobs submitted at the same time from being assigned to the same GPU. Note that technically, the queuing system does not do this, but the first part of the template script that does do this has to be run while another GPU job has actually started. If two template scripts run at the same time, they will use the same rules and select the same GPU.

The template submission script mentioned includes a section to automatically select an appropriate set of GPUs. This can be bypassed if desired, selecting a single GPU (or set, see section below) with

export CUDA_VISIBLE_DEVICES=1

For the purposes of the running job (or bash shell session), the only GPU that will be available is number 1. However, for the purposes of selecting a GPU to use while running a program (like GROMACS) from this shell session, the GPU that should be selected is 0. This is because there is only one GPU that is visible, and so it gets indexed to 0. If two GPUs were selected (regardless of their number), they would be 0 and 1 for the purposes of the shell session.

Running AMBER Simulations[edit]

To run a single CPU simulation: 

 pmemd -O -i mdin -o mdout -p ...

The above starts a standard, 1 CPU job.

For multi-CPU: 

 mpirun -np 16 pmemd.MPI -O -i mdin -o mdout-p ...

The above command uses mpi to run a job using 16 logical cores. Note, however, that the SGE queuing system is not set up to in any way to monitor and restrict the number of CPUs being used. This is technically easy to set-up, but the primary purpose of this node is GPU runs. With that in mind, all CPUs are left free to be used by any running GPU process that needs them for support. When using GROMACS, this means that CPU resources must be carefully allocated by the user when making the command line call for mdrun rather than relying on the queuing system.

To run a single GPU simulation: 

 export CUDA_VISIBLE_DEVICES=0
 pmemd.cuda -O -i mdin -o mdout -p ...

Note that first the GPU to use is selected. Here, the GPU with id 0 is being used. Then, the simulation may be run. Note that AMBER just checks this environment variable to figure out which GPU(s) it should pin the simulation to. One could submit the job above, then change to another directory, change CUDA_VISIBLE_DEVICES to, say 1, and re-run the job. However, if the template submission script is used, the GPU to use should be pseudo-optimally selected anyway. If the command

nvidia-smi -c 3

is executed, the GPUs will all be put into exclusive mode. This allows pmemd.cuda to automatically select a GPU to use, but prevents running of multi-GPU jobs. Do NOT change this setting - instead try and work with the SGE queuing system via the command line or the GUI available through the qmon command.

To run a multi-GPU simulation: 

 export CUDA_VISIBLE_DEVICES=0,1,2,3
 mpirun -np 4 pmemd.cuda.MPI -O -i mdin -o mdout -p ...

This sets 4 GPUs to be visible (should all be peer-to-peer enable with each other) and runs a simulation using all of them.

Restarting Syrah[edit]

When restarting Syrah, nothing should need to be done. However, the SGE queuing system may need to be restarted. Information may be found here.

Essentially: log in as the administrative user for the grid engine (exx on Syrah) and execute 

 cd /usr/local/ge2011.11/default/common
 ./sgemaster start
 ./sgeexecd start

Additionally, in order to ensure that nvidia-smi runs smoothly, execute the following: 

nvidia-persistenced --persistence-mode

Alternatively to the above command, reboot Syrah with a monitor connected (a Syrah-compatible monitor should be in the office or machine room. Plug it into the 3rd DVI port from the right).

Connecting Syrah to the Network[edit]

Syrah is connected to Zin's local network via an ethernet cable to the cluster network switch. However, this does not immediately make it accessible from the cluster frontend. The /etc/hosts.local file needs to be modified in Zin to include the following

10.255.255.230 syrah.local syrah

After this modification, you need to run

rocks sync config

This arbitrarily assigns the local address of 230 to Syrah. Any number within the range 1 to 254 will work here as long as it isn't already taken by one of the compute nodes. Note that whatever you select should be consistent with the local IP address (local netmask ID) you assign on Syrah itself.

Syrah is also connected to the campus network via a second ethernet cable. Commands for configuring both these networks are below: 

nmcli con mod enp13s0f0 ipv4.addresses 128.111.174.85/29
nmcli con mod enp13s0f0 ipv4.gateway 128.111.174.81
nmcli con mod enp13s0f0 ipv4.dns “128.111.1.2,128.111.1.1”
nmcli con mod enp13s0f0 ipv4.method manual
nmcli con up enp13s0f0
nmcli con mod enp13s0f1 ipv4.addresses 10.255.255.230/8
nmcli con mod enp13s0f1 ipv4.gateway 128.111.174.84
nmcli con mod enp13s0f1 ipv4.method manual
nmcli con up enp13s0f1

In order to ensure the correct routing from the campus network, create a file named /etc/sysconfig/network-scripts/route-enp13s0f0 with the line: 0.0.0.0/0 via 128.111.174.81 dev enp13s0f0.

To set up the firewall using firewalld, create a new zone with limited ssh access and remove ssh as a service from the public zone: 

firewall-cmd --permanent --new-zone=ssh_limited
firewall-cmd --permanent --zone=ssh_limited --add-source=128.111.0.0/255.255.0.0
firewall-cmd --permanent --zone=ssh_limited --add-source=169.231.0.0/255.255.0.0
firewall-cmd --permanent --zone=ssh_limited --add-service=ssh
firewall-cmd --permanent --remove-interface=enp13s0f1
firewall-cmd --remove-interface=enp13s0f1
firewall-cmd --permanent --zone=ssh_limited --add-interface=enp13s0f1
firewall-cmd --zone=ssh_limited --add-interface=enp13s0f1
firewall-cmd --permanent --remove-service=ssh
firewall-cmd --reload
firewall-cmd --runtime-to-permanent

You may need to restart the network at some point for some of the configuration to work: systemctl restart network.
