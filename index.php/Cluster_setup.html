User management[edit]

Adding a new user[edit]

  • 1) Run: 

system-config-users

This brings up a GUI, which can then be used to add a new user. Usernames are typically in the format <xy> where x = first initial and y = surname (unless the user requests otherwise).

  • 2) The GUI-default home directory for the user is the physical RAID location /export/home/<username>. Change this to point to the NFS mount i.e., /home/cask0/home/<username>.
  • 3) UNSELECT the "Create private group for the user" checkbox. All users (unless otherwise required), belong to the common group called (creatively enough!) "users". Let the system select the user id automatically. User ids are usually selected in integer increments of 1.
  • 4) Close the GUI and update the user database on all nodes by running: 

rocks sync users

Setting up a wiki account for new users[edit]

The easiest way to do this is to access the wiki through a web browser and log in as a system administrator. There is only one system administrator, called WikiSysop. At the wiki login page, type WikiSysop as the username and then the password found in LocalSettings.php under wgDBpassword (this file may be found on the cluster under /var/www/html/wiki). Once logged in, navigate to "Special" pages (on the left under "Tools"). Once here, click the link to "Create account," and follow the steps there. You only need to provide a username and password, as an email address, etc. is not required. For convenience, it is probably a good idea to just use the same username as on the cluster. However, the passwords for the wiki are perhaps not stored as securely as those for the user accounts on the cluster, so it is probably a good idea to use a different password for the wiki.

Maintaining a test user[edit]

Always maintain a dummy user profile. Right now there's one called testuser. This is useful for testing user management protocols, and also useful for logging into the cluster in case you messed up and locked yourself and others out.

Cluster structure[edit]

Hardware[edit]

The Shell group cluster is a 18 node cluster each with 8 cores, so that you have a total of 144 cores. Out of these, 8 cores belong to the head-node identified as "zin" (network name: zin.cnsi.ucsb.edu) that is used to log into the cluster. This is also the only node that has an access to the outside world. Compute nodes are named as "grape-x-y" where x = 0,1 y = 0 to 7 for x = 0, and y = 0 to 9 for x = 1. The remaining node is called "cellar-0-0" and is dedicated to managing all the user data kept on the RAID array. The hard-disk in cellar-0-0 has a partition where the RAID array is mounted. In the physical cluster (Elings Hall, 3rd floor), the RAID array is a 14 TB storage bay kept at the base of the cluster, right above the UPS (uninterruptible power supply) that has redundant disks, which house the data in a distributed fashion and provides fault-tolerance to disk failures. Right above the RAID array sits cellar-0-0, followed by zin. Next up are the grape-0-{0 to 7} followed by the grape-1-{0 to 9} nodes. Syrah is a standalone node with an independent operating system. It is not controlled by Rocks management software and as such is described in a separate section of the wiki.

The UPS was replaced in 2018. Details of the UPS can be found here.

The connections to the mains is explained here. It is recommended to not have the mains directly connected to the RAID array, but through the UPS, since sudden power surges during emergency or planned shutdowns can corrupt data.

Operating system[edit]

The cluster runs Rocks 6.2 Sidewinder, which is a server-compatible version of CentOS. Further details can be found below.

NFS mount[edit]

Although the physical location for all the data is within the RAID array, it is accessed through a hard mount to /share/apps, called the Network File Storage (NFS). Similarly, user home directories are mounted to /home/cask0/home. DO NOT unmount these locations. A brief description of the folders of the NFS mount in /share/apps is given below:

  • scripts - all group code (python, fortran, bash, tcl)
  • installation_scripts - whenever you install recent software, try to compile a list of instructions and put it in a file under this. You can dump any additional things you want into that file or folder. Files under this directory should be named as get_software.sh. Update the README if required. If a software has plugins built by the group that need to be put in at compile time, you’ll find them here.
  • admin_scripts - treat all code in here with EXTREME CAUTION. Blindly executing any of these shell scripts can break your or others' work instantly. The scripts here are useful for propagating changes manually to compute nodes if they are rebooted or reinstalled. Contact M. Scott Shell if you don’t understand something in here. Bottom line: Do not run anything from here without prior confirmation. (If there is still confusion, shoot an email to tanmoy.7989@gmail.com)
  • package/version - softwares that you install should go into their own folders and wherever possible try to stick to that <package>/<version> hierarchy
  • modulefiles - whenever you install some new package/version, write an appropriate modulefile for the same and keep it under modulefiles/package/version. If you built only a new version of an already existing software, remember to keep it under the appropriate package location in modulefiles
  • src - store any necessary software sources here. Try and make a new folder for new sources. Sources mean only zipped files or tarballs.

Common admin tasks[edit]

All admin tasks may be resolved by consulting the Rocks User Guide. Consider the below as a quick reference, but if you're not sure about anything, read the manual.

Monitoring load[edit]

The I/O or computational load on the cluster can be monitored through the ganglia tool. High usage spikes for cellar-0-0 reflect users archiving data. Whenever you see I/O is slow, this should be the first thing to check.

Login script[edit]

The script /share/apps/login.sh contains all path-names, environment variables and module loads that may be useful for all users all the time. The important environment variables to take care of are: 

export APPS=/share/apps
export PATH=$APPS/scripts:$PATH

Further, the default temp directory (where temporary files are created by the OS and programs during run-time, and are flushed out periodically) is /tmp which has limited space. So make sure that the default is changed to the more spacious /state/partition1/tmp: 

export TMPDIR=/state/partition1/tmp

Modules that MUST be loaded at login-time include the C/C++/Fortran compilers. For other software/libraries, decide based on how frequently they are going to be used and by how many group members.

Shutdown[edit]

For planned shutdowns, send an email out to all users to kill / pause their jobs prior. Then (as root), run 

rocks run host command="shutdown -h now"

Restart[edit]

If the entire cluster is rebooted after shutdown, the head-node and cellar-0-0 does a simple reboot (like how your laptop works), whereas the compute nodes can be customized to do either a reboot or a "re-install". To force a re-install on a compute node (say grape-x-y), run the following before you shut down the cluster: 

rocks set host boot compute-x-y action=install

Don't set this action unless necessary, e.g. when the grape-x-y node was misbehaving prior to shut-down and you didn't know how to fix it. This can also be done to fix a particular compute-node if it is doing something weird that you don't know how to fix. Simply set it to re-install the OS, then shut it down and bring it back up. NEVER, EVER, EVER set this action for the head node or cellar-0-0, else you'll lose all data. When restarting the entire cluster, follow this order: 

zin --> cellar-0-0 --> grape-0 nodes --> grape-1 nodes

Wait till zin reboots completely, then start cellar-0-0 and wait till it reboots completely. After that you can reboot all the compute nodes one after another.

Post-install tasks[edit]

  • When a compute node is rebooted / reinstalled, it should automatically run all post-install stuff as detailed in /share/apps/admin_scripts/extend-compute.xml. ONLY IF IT DOESN'T, run /share/apps/admin_scripts/manual_postinstall/buildnodes.sh as root.
  • If you reinstall cellar-0-0 (as mentioned before, DON'T), be careful with the root partition formatting, and DO NOT TOUCH /dev/sdb1 - this is the RAID. The extend-nas.xml script kept in /share/apps/admin_scripts is automatically added to the OS and should take immediate effect. ONLY IF IT DOESN'T, ssh to cellar-0-0 and run the commands in /share/apps/admin_scripts/manual_postinstall/hardmount.sh. Run the bash commands in this scripts one by one, NEVER at one go.
  • When the head-node is rebooted after a shutdown, there is a bug in the Rocks OS that may sometimes prevent the ganglia tool to recognize the compute nodes and the ganglia webpage will show the head-node as the only node on the cluster. In this case, run /share/apps/admin_scripts/fixganglia.sh as root.
  • If for some reason you need to reinstall (not reboot) the head-node (again, DON'T), run the following scripts in the order given: 

bash /share/apps/admin_scripts/manual_postinstall/buildzin.sh
bash /share/apps/admin_scripts/manual_postinstall/buildnodes.sh
bash /share/apps/admin_scripts/fixganglia.sh

Look at what the commands in buildzin.sh do; chances are you might not need to run all or most of them. More details on the Rocks specific commands in these scripts can be found on the online Rocks User Guide.

Installing new software[edit]

Sometimes common Linux packages such as libraries, etc that need installation can be already present as RPMs in the Rocks OS. In those cases search the Rocks rpm repository using: 

rpm -qip <package>.rpm

Or via yum: 

yum list <package>

If this gives any output, then the RPM is probably present already and all you need to do is: 

rpm -i <package>

One can also use yum to perform the installation if the RPM is available via: 

yum install <package>

If the RPM is not already available, you can download the appropriate RPM and place it in /export/rocks/install/contrib/6.2/x86_64/RPMS as described in the Rocks User Guide section 5.1. To make the package available (to any node), you just have to then run 

cd /export/rocks/install
rocks create distro

To then install the package on a specific host (i.e. grape-0-0, cellar-0-0, frontend, etc.) you can run 

rocks run host <hostname> "yum clean all; yum -y install <package>"

From the user guide, you should be able to figure out a variant of this command in order to install on all compute nodes, etc. Note that you DO NOT have to add the package to the extend_compute.xml or extend_nas.xml files in /export/rocks/install/site-profiles/6.2/nodes. You only need to do this if you want that package to automatically be installed when the node is re-installed, which is probably a good idea, but your call. However, this also means that you can install packages on individual nodes without having to set them to be re-installed and re-boot them. This is a very nice feature when working with cellar, which is a NAS appliance node, where re-installing the node is risky.

For the purposes of record keeping, challenges faced when installing GCC 7.0 and LAMMPS are outlined below:

  • GCC - Rocks 6.2+ comes with a HPC roll which must be installed to automatically get Sun Grid Engine. It also gives a openmpi. Great, right ? Well no, because the gcc version that comes with CentOS is old but needs to be kept for compatibility purposes. That means you need to start installing new software after installing gcc. Please install all other software with this C compiler. Just run module load gcc/gcc-<version> before starting any other install. An advisable flowchart is

gcc (comes with g++, gfortran) → anaconda python → pip → mpi→ fftw→ BLAS→ LAPACK→ Lammps → Gromacs → Amber→ packmol→ pymbar→ others Optional: intel parallel studio (free version, NOT INSTALLED currently). If you do this, you will have to go back and install separate versions of mpi compiled with the intel compiler as well as all softwares that use mpi. Remember to create separate module files in each case. It is easy to run into a large number of permutations, so think carefully what compiler-mpi-software combo you need and only install those.

  • LAMMPS - Making LAMMPS may be slightly non-standard on the cluster, since the Makefile needs the location of fftw libraries (among other things), which you need to specify. Currently an example makefile is kept in /share/apps/lammps/lammps-15May15/src/MAKE/Makefile.mpich2. Follow this when installing other versions in future. The fftw library is presently located in /share/apps/fftw which is why "include" and "lib" folders under this fftw location must be supplied to the FFT_INC and FFT_PATH locations in the Makefile. These locations would obviously change if the fftw libraries were located differently. Leaving these fields empty in the Makefile will force LAMMPS to install its own KISS fft library which is relatively much slower.

Finally, as a general set of guidelines:

  • Keep all software sources (wherever possible) in /share/apps/src. (delete all untarred or unzipped stuff from here)
  • If you can, write down the bash commands for the installations into a single script, with appropriate documentation and place it in /share/apps/installation_scripts
  • If multiple versions of this software exist on the cluster already, PLEASE create a modulefile under /share/apps/modulefiles/. Modulefiles follow a c-shell (as opposed to the bash shell) format; look at the other modulefiles in this directory to understand how to write one. If (and only if, discuss with M. Scott Shell first) the version of the software you installed can be used system-wide, go ahead and put a line: module load <modulefilename without path> in /share/apps/login.sh.

Configuring the firewall[edit]

The default rocks firewall should mostly be fine. To additionally only allow ssh access from on-campus or from VPN connections to the campus network, use the following commands 

rocks remove firewall global rulename=A20-SSH-PUBLIC

rocks add firewall global network=public protocol=tcp \
service=ssh chain=INPUT action=ACCEPT \
flags="-m state --state NEW --source 128.111.0.0/255.255.0.0" \
rulename=A20-SSH-PUBLIC

rocks add firewall global network=public protocol=tcp \
service=ssh chain=INPUT action=ACCEPT \
flags="-m state --state NEW --source 169.231.0.0/255.255.0.0" \
rulename=A20-SSH-PUBLIC-VPN

rocks add firewall host=localhost network=public protocol=tcp \
service=www chain=INPUT action=ACCEPT \
flags="-m state --state NEW --source 128.111.0.0/255.255.0.0" \
rulename=A40-WWW-PUBLIC-NEW 

rocks add firewall host=localhost network=public protocol=tcp \
service=www chain=INPUT action=ACCEPT \
flags="-m state --state NEW --source 169.231.0.0/255.255.0.0" \
rulename=A40-WWW-PUBLIC-NEW-VPN

rocks sync host firewall localhost

Note that this only configures the firewall for the frontend, even though the global firewall is changed, which applies to all nodes. However, since only the frontend (localhost) is connected to the public network through eth1, the other nodes will just ignore this command. They will get updated when the nodes are restarted, but it won’t really matter. The last two commands allow zin.cnsi.ucsb.edu to be accessed from a web-browser as long as you are on campus or connected via a VPN. Additionally, they magically cause the cluster wiki and ganglia to start working as well, which I guess makes sense because you access both these tools by going through zin.cnsi.ucsb.edu or zin.cnsi.ucsb.edu/wiki.

In order to allow access to specific IP addresses (note: this is only relevant if the individual needing access cannot obtain a UCSB netID, so an unusual case), run a similar command, replacing IPADDRESS with the IP address and RULENAME with a label 

rocks add firewall global network=public protocol=tcp \
service=ssh chain=INPUT action=ACCEPT \
flags="-m state --state NEW --source IPADDRESS" \
rulename=RULENAME
rocks sync host firewall localhost

To remove the rule, run 

rocks remove firewall global rulename=RULENAME

Reinstalling compute nodes[edit]

If a compute node is behaving strangely, try first forcing a re-install by running 

rocks set host boot grape-x-y action=install

(Note: do NOT set this action for the head node or cellar.)

If that does not work (e.g. there is a filesystem error in /state/partition1, which is preserved over reinstallations), then you can try removing the node from the cluster and adding it back. To do this, shut down the node and run 

rocks remove host grape-x-y
insert-ethers --hostname=grape-x-y --cabinet=x --rank=y

After running the last command, select Compute and then turn on the node. Enable a PXE boot on that node (F12). Wait until you see an asterisk (*) within the parentheses (as described in the Rocks User Guide section 3.3), indicating that the compute node has successfully requested a kickstart file from the frontend. Now, quit insert-ethers with F8 (this may not be necessary if you used the hostname option with insert-ethers). This will perform a full reinstall of the compute node. You will likely have to wait about an hour for the compute node to show up on the queue.

Build the cluster from scratch[edit]

Authors: Tanmoy Sanyal, Jacob I. Monroe, Sally Jiao

The cluster runs Rocks 7.0 (Manzanita) which is a server-compatible version of CentOS 7.4. This page describes the steps involved in upgrading the cluster from one Rocks version to another, based on experiences upgrading from Rocks 4.3 to 6.2 in 2017 and from Rocks 6.2 to 7.0 in 2023. The steps outlined here assume that the RAID is not being reconfigured. If you are also doing this, see Building/Rebuilding/Reconfiguring the RAID and simply add in those steps after installing the NAS node, but before mounting the RAID.

Some helpful resources are the Rocks documentation (here for 7.0), the NPACI Rocks mailing list, Rocks Clusters forum on Google Groups, and the wiki page on the rocksclusters github. A step-by-step installation of Rocks 5.3 can be found here.

Back up files from the current configuration[edit]

Usually the best thing to do is to create a restore roll. For an overview of how to create and use a restore roll, check out the documentation here. For more (necessary) details, check out this page.

We did not use a restore roll during either of the upgrades. In 2017, there were "weird errors" when attempting to create it. In 2023, it was created but ultimately not used (using it after installing the compute nodes resulted in strange errors).

Instead, we backed up the following:

  • Screenshot of the window with user names and ID numbers that pops up when you run system-config-users. The ID numbers are supremely important for restoring data later on.
  • User data in /home/cask0/home
  • Everything in /share/apps/admin_scripts
  • Everything in /share/apps/installation_scripts
  • /share/apps/login.sh
  • /etc/profile
  • /etc/bashrc
  • /etc/vimrc
  • /etc/hosts
  • /etc/hosts.local
  • /etc/httpd/conf/httpd.conf
  • /var/www/html/wiki/LocalSettings.php

To be safe, it is also a good idea to make tarballs of /share/apps, /etc, /var and the wiki. Backing up the wiki is described here.

Also, make a test user account (or, make sure there is an existing test user account with a few files) to test user creation and file ownership in the new configuration.

Build compute nodes[edit]

1) Open a terminal on the head node and run: 

insert-ethers

Choose appliance type as “Ethernet Switches” and wait until it catches the signal from the network switch. (Note: The network switch should be turned on.) This can take anywhere between 1 to 10 mins. Once the network switch is identified it gets registered with name network-0-0. Now, quit insert-ethers using F8.

2) Now type: 

insert-ethers --basename=grape

and switch on the compute nodes starting from the bottom to the top of the bottom rack. Do not turn them on randomly; follow bottom-to-top order and wait for each node to catch the signal from the network switch. Note that you may need to run service httpd start and wait a minute or so before the insert-ethers command works. You also may need to enable PXE boot on the compute nodes (hit F12 after rebooting). You can monitor the installation using: 

rocks-console <compute-node-name>

where name = grape-0-{0-7} or grape-1-{0-9}.

For the some of the nodes, Rocks 7.0 may request that the disks be manually partitioned (similar to the head node). Follow the same partitioning as for the head node.

Once all bottom rack (grape-0) nodes have been installed and rebooted to show the login prompt, quit insert-ethers using F8.

3) Type 

insert-ethers --basename=grape --cabinet=1

and switch on the top rack nodes, strictly in the order of bottom to top, one by one. Again, monitor Rocks installation on each of them and wait till all of them rebooted to a login prompt. After that, quit insert-ethers using F8.

At this point, you will be able to ssh back and forth between the head and compute nodes. However, according to Rocks’ default policy, all the compute nodes have a random root password and so you will not be able to login as root from the compute node terminal. To have root privileges on a compute node, login as root to the head node and ssh to the compute node.

Build the NAS-node (cellar-0-0)[edit]

Be careful when handling cellar, as it is easy to accidentally erase data in the RAID array from here. Use the testuser account to play with and identify any potential data loss. (Also discussed here)

1) Install the nas-node with 

insert-ethers --basename=cellar

and choose appliance type as "NAS Appliance" on the GUI that opens from the head node. Enable a PXE boot on the storage node upon booting (F12). Partition the disk following the same partitioning as the head node except: DO NOT TOUCH sdb or format it. That is the RAID. (“Software comes and goes, data is forever” - some guy at San-Diego-Supercomputing-Center)

2) Then let cellar-0-0 reboot to the new Rocks version. At this point cellar sits with the partitions created above and all the RAID data present in /dev/sdb1. You can check by mounting /dev/sdb1 using mount /dev/sdb1 /some/mnt/point and /some/mnt/pnt/home will have all the user data, each owned not by a user (because there are none!) but user IDs that go up from 1000 onwards. Again, It is recommended that prior to cluster upgrade, you have a list of all the usernames and IDs and a dummy test user to test user creation.

Mount RAID on cellar[edit]

Mount through xml extensions[edit]

1) Make a copy of the extend-nas.xml file currently present in /share/apps/admin_scripts, and add these lines (IF NOT ALREADY PRESENT) right before the line at the bottom that should read </kickstart> 

<post>
<eval shell="bash">mkdir -p /export/raid</eval>
<file name="/etc/fstab" mode="append">
/dev/sdb1               /export/raid            xfs    defaults        0 2
</file>
<file name="/etc/exports" mode="append">
/export/raid 10.0.0.0/255.0.0.0(rw,no_root_squash,async)
</file>
</post>

2) Transfer this copy of the extend-nas.xml file to /export/rocks/install/site-profiles/<version>/nodes, where <version> is currently 7.0.

3) Then, from /export/rocks/install, run: 

rocks create distro

This will bind these instructions within the Rocks distro. Now, reinstall cellar.

WARNING: At this step, I messed up (deviated from the instructions given above) and got locked out from the cellar-0-0 node. So, I did a manual RAID mount at this point, the details of which are given below.

Mount manually[edit]

1) Create the /export/raid directory with mkdir -p /export/raid.

2) Manually edit /etc/fstab and add: 

/dev/sdb1               /export/raid            xfs    defaults        0 2

3) Manually edit /etc/exports and add: 

/export/raid  10.0.0.0/255.0.0.0(rw,no_root_squash,async)

4) Then run: 

mount -a
exportfs -ra

for the changes to take effect immediately on cellar. Test that the nas mount went through by creating some arbitrary /mnt/point on head node and running mount cellar-0-0:/export/raid /mnt/point and check that you can see all the user data in /mnt/point/home. If you want you can also create a /export/data1 directory since Rocks automatically treats any dir called /export/data1 on the nas as mountable remotely.

Mount RAID on all nodes[edit]

1) On the head node, add 

cask0  cellar-0-0.local:/export/raid

to the file /etc/auto.home. This will take effect when you restart the head node, but in this age of instant gratification, put it into effect immediately by running: 

/sbin/service autofs restart

2) Migrate the change to all compute nodes using Rocks' 411 service: 

make -C /var/411 force
rocks run host command=’411get --all’
rocks run host command=’/sbin/service autofs restart’

All of this has been encapsulated into the bash script /share/apps/admin_scripts/manual_postinstall/hardmountraid.sh. However, the xml extension approach is preferred since building extend-nas.xml directly into the Rocks distro makes all these steps automatic anytime there is a reboot.

Add users[edit]

It is highly recommended that right before an upgrade, you create a dummy user (say, testuser) with some dummy password, on which you can run all these steps and make sure it works before adding real users.

1) Add users by running system-config-users. When you add users, make sure their home directory is /home/cask0/home/<username> and their user ID is the same as their user ID in the previous cluster configuration (this will preserve their ownership over their files). There were some upgrade-specific quirks:

For the Rocks 6.2 installation: Even with proper care to preserve all user metadata from previous cluster configuration, all users will be shown to be “nobody”. Without going into too many technicalities, the source of this problem is conflicting user metadata (so called nfs id maps) that ROCKS tries to salvage from the RAID. To remove this cached metadata type nfsidmap -c. This deletes any cached (and incorrect) metadata. More can be found here. NEVER do this to an active and healthy cluster. It will abolish everybody's ownership of their data.

For the Rocks 7.0 installation: User IDs below 1000 are now discouraged (previously, user IDs went from 500 upwards). We made users with their user IDs from the previous configuration and then ran usermod -u NEWID USERNAME to change their IDs to 1000 and above. This also takes care of transferring file ownership in the home directories to the new user ID.

2) Then run rocks sync users to update the user database on all nodes. Give the change several minutes to take place for each user.

Modify user login details[edit]

A file called login.sh (keep a backup from the current system) is to be kept under /share/apps. This controls all the file paths, environment modules etc for all users.

1) At the end of /etc/profile add the line 

source /share/apps/login.sh

2) Make a copy of the extend-compute.xml file in /share/apps/admin_scripts and add the following lines (IF NOT ALREADY PRESENT) before the line </post>: 

<eval shell="bash">
mkdir -p /state/partition1/tmp
</eval>

<file name="/etc/profile" mode="append">
source /share/apps/login.sh
</file>

The first set of lines changes the system temporary directory to /state/partition1/tmp (more details of why this is done can be found here.) The second set of lines adds the common login details to all compute nodes.

3) Put this new version of the extend-compute.xml file in /export/rocks/install/site-profiles/<version>/nodes where <version> is currently 7.0. Then from /export/rocks/install, run 

rocks create distro

to bind it into the current rocks distribution.

4) Finally reinstall the compute node/(s). Remember reinstalling and rebooting a computing node is different (more details here). To reinstall, type 

rocks set host boot grape-0-0 action=install

and then physically reboot the node. Again, as before with xml script extensions are the best approach since they make all these post-installation changes a part of the Rocks distro and thereby automatic. However, these instructions were collected and also placed in the bash script /share/apps/admin_scripts/manual_postinstall/buildnodes.sh, which can be used to run these post-install protocols manually, when a compute node is reinstalled.

Create a queue[edit]

1) If needed, start SGE by running /etc/init.d/sgemaster.zin start

2) Make sure sge_execd is running on all compute nodes. If it is not, run: 

rocks run host compute “/opt/gridengine/default/common/sgeexecd start”

3) Add all compute nodes to the list of hosts by running qconf -mhgrp @allhosts

4) Add the number of slots available in each compute node to the queue configuration by running: qconf -mq all.q. Modify "slots," which should be a list that looks like 1,[grape-0-0.local=8],[grape-0-1.local=8] etc. (including all the compute nodes). 8 is the number of slots per node.

5) In order to allow nodes to send job status emails, you must install mailq on the compute nodes: 

rocks run host compute "yum clean all; yum -y install mailx"

RAID array[edit]

Hardware and configuration[edit]

The RAID array is housed in a Dell Powervault MD1000. This comes with a PERC 5/E RAID controller, which can handle 15 hard-drives, each up to 2 TB in size (note that hard drives that sell as "2 TB" are actually only around 1.8 TB) (other controllers, such as the PERC H800 can handle up to 4 TB drives). Available options for RAID levels may be found in the manual, though RAID 10 is strongly recommended. While RAID 5 provides more space for the same number of drives, it is highly error prone once the size of each size becomes large (say bigger than about 1 TB). The RAID has fourteen 3 TB drives in a RAID 10 configuration, providing 14 TB of storage due to the limitations of the PERC 5/E controller. If the full amount of each drive could be utilized, the size of the RAID would grow to around 19.6 TB (drives sold as "3 TB" are actually only around 2.8 TB). Note that in a RAID 10, each drive has a partner that mirrors its data. This means that you can lose half of your drives as long as you don't lose two partners, and if you do lose a pair you will only lose part of your data, not all of it, as with a RAID 5. This also means that the RAID must be expanded by adding pairs of drives. In other words, you always have to have an even number unless you have a hot spare. A hot spare is a drive that sits spun up on the storage array, but does not hold any data. If any other drives fail, the hot spare immediately takes its place. Once the failed drive is replaced, it becomes the hot spare. A hot spare is a nice system to have in place, and should be implemented if more drives are available. However, the main advantage of a hot spare is that it minimizes cluster down time, not necessarily improved data security. Without a hot spare, the cluster will be back online as soon as the failed drive is replaced, which will likely take no more than a day.

Building/Rebuilding/Reconfiguring the RAID[edit]

BEFORE STARTING, MAKE SURE ALL DATA IS SECURELY BACKED UP. This means asking all users to back up their own data, then making a backup of /home/cask0/home. You will need this backup to restore from and the user's individual backups if anything goes terribly wrong.

First, you need to reset the boot instructions for cellar so that it doesn't try (and fail) to find a formatted, mountable partition on the RAID array. Unless you're doing minor reconfigurations of the RAID, you will need to manually create a file system and partition, as described below, before trying to mount them. If you do not disable these boot instructions, boot will fail and require a root password to login and set things up. Unfortunately, the default ROCKS behavior is to create a random password on every node that is not the head node, so you won't know the root password on cellar. The way to manage any node as root is to login to the head node as root, then ssh into the node you want to work with. This preserves security, so it is strongly advised that the individual node passwords remain random.

Now, back to business - you want to essentially reverse the instructions in Mount RAID on cellar. Make sure to make a copy of extend-nas.xml, then in /export/rocks/install/site-profiles/6.2/nodes/extend-nas.xml remove the lines at the bottom from <post> (inclusive) to </kickstart> (not inclusive). Then change directory to /export/rocks/install and run rocks create distro. You can then either manually go in and unmount the RAID array on cellar, or attempt to force a PXE boot upon restart of cellar. If the PXE boot doesn't work (you'll still see errors) run rocks set host boot cellar-0-0 action=install to make sure that cellar will reinstall upon boot and that it will follow the instructions in extend-nas.xml. Just be careful when reinstalling that you manually partition everything according to the directions in Building the NAS-node (cellar-0-0) (an example of what this looks like).

Before working with the RAID array, it's probably a good idea to power down the entire cluster, including cellar. This is not technically necessary if you have appropriately unmounted it, but is best practice. Next, you will want to reboot cellar and on the load screen (an example of what this looks like) hit ctrl-R to go into the RAID BIOS menu. In this menu, select the PERC 5/E controller ONLY - don't touch the other one. If you're rebuilding or reconfiguring the RAID in a major fashion, highlight the RAID virtual drive using the arrow keys (an example of what this looks like), then hit F2 to open a menu and delete this virtual drive. (Note: you may be able to use MegaCli for RAID reconfiguration. However, it does not seem to work for a RAID 10 configuration.) You can now exit the menu, then power down cellar. Next, power down the RAID array using the two switches in the back.

If you're swapping out drives, a useful order is to remove screws, pull off the SATA adapter plate, remove the drive, put the plate on the new drive, put the drive in the carrier, and then put the screws back in. Helpful diagrams are provided in the manual for the Dell Powervault MD1000.

Once you're done physically adjusting drives, power the RAID back on and reboot cellar, entering the RAID BIOS menu with ctrl-R. There should now be no virtual disks, so create a new one. Use tab to navigate between fields (helpful tip - use the spacebar to select physical drives). Note that for RAID 10, you need an even number of drives, unless you have a hot spare. Also note that by default the drive size will set itself to the maximum possible. When done, hit "ok" to create the drive. You then need to highlight the drive and hit F2, selecting "Initialization." This process will take approximately 13 hours for 14 drives).

When initialization is complete, exit the RAID BIOS menu (press esc until you get an option to quit) and restart cellar with ctrl-alt-del. Also restart the head node at this time - you will need it running in order to login to cellar as root. Remember, if cellar doesn't boot (cellar booted successfully if you end up with a login screen without errors) due to an inability to locate /dev/sdb1 (the RAID array), you may need to force a reinstall as described above.

Once cellar is started successfully, login to the head node as root and ssh into cellar. Running fdisk -l will show you that the RAID is recognized on /dev/sdb, but you now need to create a mountable partition with a file system. From the command line, run 

parted /dev/sdb

Within the parted command line, run the following commands: 

mklabel gpt
mkpart primary xfs 1MB -1
quit

This sets up a GUID partition table in the first command and creates a new partition in the second that expects an xfs file system and will start the partition at the first megabyte and end it at the very end of the drive (-1). Apparently, specifying a 0 instead of 1 MB is not optimal and parted will complain.

Now the partition will be created (when you run fdisk -l you will now see a partition called /dev/sdb1), but we need to actually put a file system on it. This can be done from the command line with the following 

mkfs -t xfs -d su=64k -d sw=5 /dev/sdb1

If you get an error mkfs.xfs: No such file or directory, then install xfsdump with the following and then rerun the mkfs command above. 

rocks run host cellar-0-0 "yum clean all; yum -y install xfsdump.x86_64"

The partition will now be ready to mount, which can be done manually as described in Mount RAID on cellar. (Note: if you get an error exportfs: Failed to stat /export/data1: No such file or directory then you need to mkdir -p /export/data1 first.)

To check that all this worked, you can run df -h. If you see /dev/sdb1 listed with the correct size and mounted at /export/raid, everything has been successful.

Back on the head node, you will also want to make sure to follow the instructions in "Mounting through xml extensions" to modify the extend-nas.xml file appropriately and rerunning rocks create distro, etc. To make sure all of this worked, you can restart cellar, even reinstalling to again make sure the appropriate commands are followed (though again, be careful with doing manual partitioning). If the RAID is now appropriately mounted on cellar, go ahead and reboot the head node as well as the other cluster nodes. With df -h you should now see an empty drive under /home/cask0/ (note that you may need to ls /home/cask0/ first because of the automount).

At this point, you can transfer data back on from your backup by mounting the external hard drive. It's recommended, for efficiency, that you connect it directly to cellar. Then, ssh into cellar from the head node and run: 

mkdir /media/externalHD
mount /dev/sdc1 /media/externalHD

As a helpful tip, create the backup with something along the lines of rsync -avhP /home/cask0/home backup_directory_name_on_external_drive, and the reverse when putting things back on. The "a" will ensure that all of the appropriate permissions are applied so everything isn't owned by root when you put it back on. In order to check and make sure everything has transferred correctly, you can run rsync -avhP --checksum --dryrun ... after the other rsync command. If that does not print any file transfers that it would make, everything has been transferred accurately and you haven't missed anything. Note that both file transfers and the checking process will take a very long time with a lot of data (approximately 0.5 TB per 6 hours).

RAID health[edit]

The primary tool for assessing RAID health, as well as managing the RAID setup from the command line, is MegaCli (download compatible with our Dell PowerVault MD1000 controller is here). By googling, you can find many sites with useful commands for MegaCli.

To generally check and make sure things are ok and check the RAID configuration, ssh to cellar-0-0 (as root) and then run: 

/opt/MegaRAID/MegaCli/MegaCli64 -AdpAllInfo -aAll

This provides information on each of the adapters that is controlling a virtual disk (or array). If you want more compact information about the virtual disks themselves run: 

/opt/MegaRAID/MegaCli/MegaCli64 -LDInfo -Lall -aAll

And for information on all physical disks, run 

/opt/MegaRAID/MegaCli/MegaCli64 -PDList -aAll

If you note the "Device Id" for each of the disks in the above command, you can further check disk health with the smartctl command line tool like: 

smartctl -a -d sat+megaraid,<Device Id> /dev/sdb1

You'll need to do this for each disk, with a lot of output generated.

Replacing drives[edit]

Sometimes a drive may fail and need to be replaced, or a drive may seem to be failing (causing lots of errors) and need to be preemptively replaced. This is pretty easy to do, regardless of whether or not we have a dedicated hot-spare (meaning we have one drive on reserve to take over in case another fails). To replace a drive, follow the directions below, or see this, for example.

First figure out which drive is bad by checking the information generated in the section above. You'll want to note the "Slot Number" of the drive to be replaced.

Next, you need to set the drive to be removed to be offline (not be used). In what is below, the zero in "[0:<Slot Number>]" is the enclosure number, which should always be zero for our RAID setup, and the "-a1" refers to the adapter, where the RAID of interest is on adapter 1. Note that this command, and the two following (marking the drive as missing and preparing the drive for removal) may not work if the drive is already in the failed or "Bad" state, which you can check with the commands above. In that case, after confirming that none of the other drives is the hot spare (again, using the commands above), you can skip the next three steps and directly remove it. 

/opt/MegaRAID/MegaCli/MegaCli64 -pdoffline -physdrv[0:<Slot Number>] -a1

After this, mark the drive as missing. The command below may or may not be successful, but try it anyway. If it doesn't work, just try the next command and hope for no errors. 

/opt/MegaRAID/MegaCli/MegaCli64 -pdmarkmissing -physdrv[0:<Slot Number>] -a1

Next, prepare the drive for removal. This command may not work, even if the first two commands worked. 

/opt/MegaRAID/MegaCli/MegaCli64 -pdprprmv -physdrv[0:<Slot Number>] -a1

When you run the above command successfully, the light above the drive to be removed should start blinking. If it does not and you want to locate the drive, you can run: 

/opt/MegaRAID/MegaCli/MegaCli64 -PdLocate -start -physdrv[0:<Slot Number>] -a1

And to stop the blinking, run: 

/opt/MegaRAID/MegaCli/MegaCli64 -PdLocate -stop -physdrv[0:<Slot Number>] -a1

Now you can remove the disk and replace it with a new one. The hardware manual is located here in case you want to feel slightly better about what you're doing, but also slightly more confused. (Note: you need to remove both the SATA adapter as well as the screws from the casing before you can slip the drive out.)

Once the new disk is inserted, you need to designate it as the dedicated hot spare: 

/opt/MegaRAID/MegaCli/MegaCli64 -PDHSP -Set -Dedicated -Array0 -PhysDrv [0:<Slot Number>] -a1

OR, if your RAID configuration supports disk mirroring, you instead will need to note that the new drive replaces the previously failed drive that was marked missing and then start the rebuilding process. (It's possible that the rebuilding process will start automatically after you put the new drive in, in which case these two commands will not work.): 

/opt/MegaRAID/MegaCli/MegaCli64 -pdreplacemissing -physdrv[0:<Slot Number>] -Array<Span Number> -row<Arm Number> -a1
/opt/MegaRAID/MegaCli/MegaCli64 -pdrbld -Start -physdrv[0:<Slot Number>] -a1

You can get Span Number and Arm Number by executing the following and looking for the slot number of the drive you're replacing: 

/opt/MegaRAID/MegaCli/MegaCli64 -CfgDsply -aALL

To check and make sure everything is alright, you'll probably want to look through information that can be generated by commands in the section above. You can also check the drive's rebuilding progress: 

/opt/MegaRAID/MegaCli/MegaCli64 -pdrbld -ShowProg -physdrv[0:<Slot Number>] -a1

Web server (i.e. this wiki)[edit]

Configuring the wiki permissions[edit]

Add the following (if not already present) to /var/www/html/wiki/LocalSettings.php: 

# Prevent new user registrations except by sysops
$wgGroupPermissions['*']['createaccount'] = false; 

# Prevent anonymous editing
$wgGroupPermissions['*']['edit'] = false;

# Prevent anonymous reading
$wgGroupPermissions['*']['read'] = false;

# Pages anonymous (not-logged-in) users may see
$wgWhitelistRead = array ("Main Page", "Special:Userlogin", "Wikipedia:Help");

Changing a user password[edit]

Enable the maintenance scripts. Then run the changePassword.php script: 

php changePassword.php --user=someuser --password=somepass

Backing up the wiki[edit]

This should be done periodically. It is also useful in that it creates a local, readable version of the wiki so that if the wiki goes down you can still reference this local version.

  1. Run wget to create a static local copy of the wiki. 
    wget --mirror --convert-links --html-extension --no-parent --random-wait --page-requisites \
--load-cookies=cookies.txt http://zin.cnsi.ucsb.edu/wiki/index.php/Main_Page

    This uses wget to recursively pull all pages connected to the wiki main page, with the exception of external links. All materials necessary to reconstruct the html pages are downloaded and linked to each other, allowing you to view the wiki without access to the internet, but using a web-browser with fully populated and connected web pages saved on your computer. All you need to do is open the 'index.php' folder with Firefox or another web browser, or open the 'Main_Page' file. The links to almost all other pages will work from this page. This is just a static reconstruction and will not contain information necessary for actually recreating the wiki with the wikitext mark-up language. This information will be collected in later steps, but here you have just created a reference for how things should look.

    Note that this requires a 'cookies.txt' file. This is what allows you to log in to the wiki and access the full content of all of the pages. To generate this file, simply log in to the wiki, then use a tool like the Firefox Export Cookies add-on to save such a file. Notice that there is not a 'cookies.txt' file currently in this back-up directory. This file must uniquely be generated before a back-up!

  2. Save a backup of /var/www/html/wiki/LocalSettings.php. This backs up preferences and settings. This is not strictly necessary, as this file may easily be set-up again through the MediaWiki installer, but it’s nice to have as a reference.
  3. In /var/www/html/wiki/LocalSettings.php, uncomment the line: 
    $wgReadOnly = 'Dumping Database, Access will be restored shortly';
  4. Run the following wherever you want the zipped back-up database to go: 
    mysqldump -h localhost -u wikiuser -p wikidb --skip-lock-tables | bzip2 > backup_name.sql.bz2

    When prompted, enter the password listed in /var/www/html/wiki/LocalSettings.php under wgDBpassword.

  5. Back up the images in the wiki. Technically, this should have already been done in step 1. However, just to be safe, run the following: 
    tar cjvf /backup_location/images.tar.bz2 /var/www/html/wiki/images/

    Change backup_location to wherever you are saving the backup.

  6. Finally, comment out the line in /var/www/html/wiki/LocalSettings.php that was previously commented in: 
    #$wgReadOnly = 'Dumping Database, Access will be restored shortly';

Restoring the wiki from a backup[edit]

The key to easily restoring is having done steps 2-4 in #Backing up the wiki, since step 1 just sets up a local image of the wiki for personal reference. More detailed directions may be found here, but the short version is presented below.

  1. If not already present, copy the files for MediaWiki 1.23 from this tar file into /var/www/html/wiki.
  2. Wipe the current database (should be named wikidb) 
    mysqladmin -u root -p drop wikidb

    The password you use for this may be found in /root/.rocks.my.cnf. The normal root password will not work here.

  3. Next, open a mysql command prompt by typing 
    mysql -u root -p

    (use the same root password as in step 1 when prompted).

  4. Create a fresh database with 
    CREATE DATABASE wikidb;
USE wikidb;
  5. If this is a cluster re-install, you will also need to create a user to manage this database. This is so that root does not need to be used all the time. This will maintain consistency with any old LocalSettings.php files, and if re-installing MediaWiki and creating a new LocalSettings.php file, make sure to set the user and password the same as found in the fields ‘wgDBuser‘ and ‘wgDBpassword‘ in the backed up LocalSettings.php file. To create this user, run: 
    CREATE USER ‘wgDBuser‘ IDENTIFIED BY ‘wgDBpassword’;
GRANT SELECT, UPDATE, INSERT, DELETE, ALTER, CREATE, INDEX, USAGE ON wikidb.* TO ‘wgDBuser‘;
quit

    When this is done, quit the mysql prompt with quit (as shown above).

  6. Finally, load in the old database with 
    mysql -u root -p wikidb < backup.sql
  7. To get things working, an update will likely be necessary. Copy over LocalSettings.php into /var/www/html/wiki, and then type 
    php maintenance/update.php
  8. If step 6 fails, or the wiki is still not visible/restored at this point, you should try and re-install MediaWiki by deleting LocalSettings.php, going to zin.cnsi.ucsb.edu/wiki in a web browser and following the steps to set-up the wiki again. After doing this, you may need to go back to step 6, or all the way back to step 1 and repeat until this step is unnecessary.
  9. Finally, copy the backed-up images into the /var/www/html/wiki/images/

Note: If you are having difficulties in setting up the database on a fresh install of Rocks (error like ‘Can’t connect to local MySQL server through socket…’), you may need to symlink /var/lib/mysql/mysql.sock to /var/opt/rocks/mysql/mysql.sock using 

ln -s /var/opt/rocks/mysql/mysql.sock /var/lib/mysql/mysql.sock

Where in the above, the second option passed is the link location and the first is the file that it will point to.

Managing the wiki from the web[edit]

You can manage many features of the wiki, like adding new users or changing user passwords, by logging in to the wiki as WikiSysop and navigating to Special pages (on the left under "tools"). Only the WikiSysop user has privileges to access and perform actions associated with the links on this page. You can login to this user with the password found in LocalSettings.php under wgDBpassword.
